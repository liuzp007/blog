开场引言  
在 Vite 项目里，“慢了就调 optimizeDeps”几乎成了一种条件反射：把某个包加进 `include`，或者把某个包塞进 `exclude`，然后祈祷问题消失。短期看，问题经常真的消失了；长期看，这种做法会把一个系统性问题变成“配置补丁集合”，最终只有少数人懂，换机器、换分支、升级依赖都会触发新的不稳定。

我想强调一个更可靠的视角：`optimizeDeps` 不是加速器，它更像一层“语义翻译与稳定化”机制。你用它的目标应该是“减少不确定性”，速度只是副产品。理解这一点，你就会更克制地介入，也更清楚介入的代价。

## 预构建不是“加速器”，而是“语义翻译层”

现代前端依赖生态里仍然同时存在 ESM、CommonJS、UMD、条件导出、深层子路径导入、动态 require 等多种形态。浏览器端的 ESM 是严格的：它需要静态可解析的依赖关系和明确的 URL。dev server 也一样，它需要稳定的模块边界来构建 module graph，并建立 HMR 失效边界。

当依赖并不满足这些条件时，Vite 必须做翻译：把不适合直接被浏览器消费的依赖变成“浏览器可消费、且可缓存”的形式。这个翻译过程通常依赖快速的打包器（常见是基于 esbuild 的转换），输出一份在 dev 场景可复用的 ESM 产物。

一旦你把预构建理解成“翻译层”，你会得到两个重要结论：

1. 预构建的首要目标是确定性：同一份依赖在不同机器、不同启动次数上应该得到相同的翻译结果。
2. `include/exclude` 的首要目标是降低翻译歧义：当自动推断不可靠时，你手工告诉系统“哪些依赖必须预构建/必须跳过”，以换取稳定。

## optimizeDeps 在 dev 中的角色：把依赖从包管理层拉到模块图

包管理器解决的是“安装与版本解析”；Vite dev server 解决的是“模块加载与运行时一致性”。这两者的边界经常被误解：以为只要 `pnpm install` 正常，dev server 就应该稳定。但事实上，dev server 的模块图构建比包管理器更严格，尤其在以下场景：

- 同一个包通过不同路径被引用（子路径导入、别名、条件导出）。
- monorepo 里存在 workspace 包的软链接与真实路径差异。
- 依赖在不同环境分支输出不同入口（browser/node 条件、development/production 条件）。
- 依赖内部使用 `require()`，对静态分析不友好。

`optimizeDeps` 的作用，是把这些“包管理层的模糊地带”尽量收敛成 module graph 的确定边界。你如果经常遇到“同一个包在页面里出现两份实例”，那不是“Vite 随机”，而是边界没有被收敛。

## 依赖扫描：入口、静态分析与“扫描盲区”

自动预构建通常从“扫描入口”开始：dev server 需要知道“页面会用到哪些依赖”。这里天然存在盲区：静态分析看不到运行时才出现的导入。典型盲区包括：

- 动态 `import()` 的变量拼接（或运行时决定路径）。
- 依赖在某个分支内才会被导入（例如按平台/按权限/按环境）。
- 通过 JSON 配置、路由表、插件系统延迟加载的模块。

当盲区存在时，你会观察到一种现象：第一次打开某个页面很慢，甚至报错；第二次打开就好了。因为“第一次访问触发了临时的转换与缓存”，但它不稳定、不可预测，并且在某些机器上可能直接失败。

这时手工 `include` 的价值不是“加速”，而是把盲区变成显式依赖，从“第一次访问的偶然行为”变成“启动时可解释的确定行为”。

```ts
// vite.config.ts（示意）
import { defineConfig } from 'vite'

export default defineConfig({
  optimizeDeps: {
    // 让扫描盲区中的依赖提前被翻译成稳定的 ESM 形态
    include: ['some-cjs-lib', 'legacy-umd-lib'],
    // 避免对带副作用的依赖做预构建（需要你明确知道为何排除）
    exclude: ['some-weird-bundled-lib']
  }
})
```

关键不在于“加了就好”，而在于：你必须能解释“为什么它属于 include/exclude”，否则这只是未来的雷。

## include/exclude 的原则：以不确定性为代价的手工干预

我建议把 `include` 看成“稳定性白名单”，把 `exclude` 看成“风险隔离名单”。决策时不要从“快不快”出发，而要从“不确定性”出发：

- 当一个依赖的入口解析不稳定（环境条件、exports 条件、子路径导入混乱），并且它是关键依赖：倾向 include。
- 当一个依赖本身已经是稳定 ESM，或者它被预构建会导致副作用重复、全局状态破坏：倾向 exclude。
- 当你无法解释它为何需要 include/exclude：先不要加，先定位根因。

一个高频误区是把 `exclude` 当成“跳过预构建就能解决一切”。实际上，exclude 往往是在承认“这个依赖不适合被 dev server 当作模块服务”，你选择把它留在原始形态，让浏览器/transform 在运行时处理。它可能暂时绕开某些问题，但也可能把问题推迟到更难调试的地方。

## 典型事故一：双份依赖与符号不一致

在大型项目里，最贵的故障通常不是“启动慢”，而是“看似正常但行为诡异”。双份依赖就是典型：同一个库在页面中出现两份实例，导致：

- React hooks 报错或状态异常（本质是两份 React 实例）。
- 单例配置失效（你以为写进 A 的配置，B 完全看不到）。
- 类型层面看起来一样，运行时却不相等（`instanceof` 失败、Symbol 比较失败）。

这类问题往往由“路径不一致”触发：一个模块通过 `node_modules` 解析，另一个通过 workspace 链接路径解析；或一个走了预构建产物路径，另一个走了源路径。你如果只用“删缓存”解决，它一定会复发。

工程上更可靠的做法是统一解析路径与依赖入口，确保同一包只有一个“真入口”。这通常涉及：

- 统一别名策略（避免同包多路径）。
- 对 monorepo 包的依赖边界做约束（不要跨包深层引用未公开的子路径）。
- 必要时对关键依赖加 include 以收敛为同一份预构建产物。

```ts
// 一个“路径不一致”的常见诱因（示意）
// A 包：import 'lib'
// B 包：import 'lib/dist/index.js'
//
// 这会让解析落到不同入口，进而触发两份实例或不同编译结果。
```

你需要把“统一入口”当成工程规范，而不是遇到 bug 才想起来。

## 典型事故二：条件导出导致的环境分裂

很多包会在 `package.json` 里做条件导出：不同环境给不同入口。问题在于，环境的判断并非总是你想象的那一个，尤其当工具链链路复杂时。

```json
{
  "name": "some-lib",
  "exports": {
    ".": {
      "browser": "./dist/browser.js",
      "default": "./dist/index.js"
    }
  }
}
```

在 dev 里你可能拿到 browser 入口，在 build 里拿到 default 入口，或者相反。于是出现“dev 正常，prod 异常”。很多人第一反应是怪 Vite，但本质是“条件导出带来的环境分裂”没有被纳入你的工程假设。

这类问题的解决，优先级顺序我建议是：

1. 先在依赖层修复：明确你要哪个入口，避免隐式条件。
2. 再在配置层收敛：通过 resolve 条件、别名或 optimizeDeps 让入口一致。
3. 最后才是补丁：在某个环境里写 hack（这通常会留下技术债）。

## monorepo 与 link：workspace 包是“源码”，不是“依赖”

在 monorepo 里，workspace 包既像依赖又像源码：你希望它像源码一样热更新、可调试；你又希望它像依赖一样稳定、单一入口。这两个目标冲突时，团队经常靠“加 include/exclude”来调参，但如果没有原则，很快就会失控。

我的经验是把 workspace 包分两类治理：

- “产品包”：对外发布、版本语义明确、入口清晰。尽量像依赖一样对待，入口稳定，避免深层引用。
- “源码包”：内部复用、频繁变更、需要 HMR。尽量像源码一样对待，保持编译一致，避免被预构建吞掉源码语义。

当你把包分类之后，`optimizeDeps` 才有稳定的决策边界：你不会一会儿把它当源码一会儿当依赖，从而让预构建与 HMR 的行为在不同机器上随机漂移。

## 反模式：把 optimizeDeps 当作补丁集合

常见误区/反模式包括：

1. 只要报错就 `include`：你会把真正的入口不一致、条件导出问题掩盖掉。
2. 只要变慢就 `exclude`：你会把预构建的稳定性问题推到运行时，制造更难复现的故障。
3. 不记录原因：没有“为什么加”的注释或文档，新人无法判断能不能删，最终没人敢删。
4. 每个包各自加：monorepo 里各个子项目随意覆盖 optimizeDeps，最后行为不一致，调试成本爆炸。

如果你只能修一条，我建议修“缺少原因记录”。因为 optimizeDeps 的配置不是“代码”，它更像“系统参数”。系统参数最怕没有解释，解释是治理的起点。

## 我的判断：optimizeDeps 应该被当作“稳定性预算”，而不是“性能按钮”

我的判断是：在成熟团队里，`optimizeDeps` 的目标是“让 dev server 行为更确定”，而不是“让某台机器更快”。你应该把每一条 include/exclude 当成一笔预算：

- 你用手工干预换来了稳定性。
- 你也引入了维护成本：升级依赖时要复核、换构建策略时要复核、迁移 SSR 时要复核。

因此最好的策略不是“尽量少配”或“尽量多配”，而是“只对不确定性最大的地方配，并且把原因写清”。当你能解释每一条配置的来源，你的 Vite 才是可治理的；否则它只是更快地把不确定性带到你面前。
