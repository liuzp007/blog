开场引言  
很多团队把 SSR 当作一个“技术选型题”：要不要 SSR、用不用某某框架、选哪个 API。真正落地之后才发现，SSR 更像一个“工程组织题”：你的代码库是否有清晰的边界，能否在 Node 与浏览器两个运行时之间维持一致性，能否把数据与缓存体系化，而不是散落在组件里。

本文的核心观点是标题那句：工程组织比 API 选择更关键。Vite 在 SSR 场景里提供的是“编译与模块加载能力”，它不会替你决定边界。你如果边界清晰，Vite 的灵活性会成为优势；你如果边界混乱，Vite 的灵活性会放大不确定性。

## 先把概念说清：SSR、CSR、SSG、ISR 与“混合渲染”到底差在哪

在工程上，渲染方式不是口号，而是影响你组织代码、组织数据、组织缓存的约束集合：

1. CSR：浏览器拿到一个壳和 JS，数据在客户端拉取，渲染在客户端完成。优势是服务端简单；劣势是首屏与 SEO 依赖额外策略。
2. SSR：服务端在请求时生成 HTML（至少生成首屏），客户端再 hydration 接管交互。优势是首屏与索引；劣势是运行时边界复杂、缓存策略复杂、成本更高。
3. SSG：构建时生成 HTML（全量或部分），请求时直接返回静态文件。优势是稳定与便宜；劣势是数据更新延迟、动态性受限。
4. ISR/增量：在 SSG 基础上引入增量更新与过期重建。优势是折中；劣势是系统复杂度上升。

所谓“混合渲染”，不是一个新的技术名词，而是“不同页面/不同片段使用不同策略”的工程选择。你可以在同一站点里：

- 列表页 SSR（首屏快且可索引），详情页 SSG（内容稳定），评论区 CSR（交互强、延迟可接受）。
- 头部/导航 SSR，3D 重组件 CSR 并延迟加载。

混合渲染的难点不在于“能不能做到”，而在于“能不能治理”：如果每个页面都随意选择策略，你会得到一个不可预测的系统；如果你把策略变成可配置、可回归、可演进的规则，你才算掌控系统。

## Vite SSR 的定位：它提供编译与模块加载，不提供一整套应用框架

Vite SSR 的一个关键特征是“开放”：它不会强迫你采用某个路由约定、某个数据获取协议、某个渲染生命周期。它更多提供：

- 在 dev 阶段：按需编译与模块加载（类似 `ssrLoadModule` 这类能力），让你能在 Node 侧直接加载 TS/JSX 源码，并保持 HMR 体验。
- 在 build 阶段：产出 SSR bundle 与 client bundle（或者仅 client），并让你能控制 manifest、静态资源映射等。

开放意味着自由，也意味着你必须自己回答这几个问题：

1. 你的“入口”是什么？服务端入口与客户端入口如何共享路由与渲染树？
2. 你的“数据层”在哪？数据在服务端取还是客户端取？如何做缓存？
3. 你的“边界”在哪？哪些模块只在服务端、哪些只在客户端、哪些必须同构？

如果没有清晰回答，SSR 会把你原本在 CSR 时可以忽略的问题全部放大。

## 关键工程边界一：同构代码不是“都能 import”，而是“都能稳定执行”

很多 SSR 故障都源自一个简单误会：同构代码不是“Node 能 import + 浏览器能 import”，而是“在两个 runtime 下执行语义一致，至少差异可解释”。常见破坏同构性的因素包括：

- 依赖浏览器全局：`window`、`document`、`localStorage`、`matchMedia` 等。
- 依赖 Node 全局：`process`、`fs`、`path`、`Buffer` 等。
- 依赖时间与随机：`Date.now()`、`Math.random()` 在 SSR 首屏会把输出变成不确定，导致 hydration mismatch。
- 依赖环境分支但分支条件不一致：例如 dev/build/ssr 的条件导入导致服务端与客户端渲染树不同。

工程治理上，我建议把代码按三个层级分区，而不是按“是否 SSR”粗暴二分：

1. `shared`：严格同构，可在两端执行，允许依赖少量纯函数库。
2. `server`：仅服务端使用，允许 I/O、缓存、鉴权等。
3. `client`：仅客户端使用，允许 DOM、交互、浏览器 API。

更重要的是制度：对 `shared` 目录要“更苛刻”，宁可少放，也不要混入运行时依赖。SSR 最怕的是“偶尔混进去一个 window”，因为它往往只在某些路径触发，很难在本地稳定复现。

## 关键工程边界二：渲染边界与数据边界必须对齐，否则缓存无从谈起

SSR 里最贵的不是渲染本身，而是“渲染依赖的数据”。如果数据获取散落在组件里，缓存就很难体系化：你既不知道数据是否可共享，也不知道是否可按路由粒度复用，更不知道服务端与客户端是否会重复请求。

一个更可治理的模式是把数据获取与渲染绑定到同一个“路由单元”上：每个路由定义一个 `loader`（或等价概念），它只负责返回渲染所需的最小数据。渲染函数只消费 `loader` 的结果，不直接发请求。

这不是为了写得“像某个框架”，而是为了让你能明确：

- 缓存 key 应该是什么（通常是路由 + 参数 + 用户态）。
- 缓存 TTL 应该如何定义（内容页长、用户页短）。
- 哪些数据允许在客户端复用（hydration 注入），哪些必须在客户端二次请求（安全或个性化）。

下面给一个极简的“路由 loader + SSR 渲染”的示意。注意：它强调边界，不追求完整框架化。

```ts
// server/router.ts（示意）
export type LoaderCtx = { url: string; headers: Record<string, string> }
export type LoaderResult = { status?: number; data: unknown; cacheTtlMs?: number }

export interface RouteDef {
  match: (url: string) => boolean
  load: (ctx: LoaderCtx) => Promise<LoaderResult>
}

export const routes: RouteDef[] = [
  {
    match: (url) => url.startsWith('/main/blog'),
    load: async () => ({ data: { kind: 'blog-list' }, cacheTtlMs: 10_000 })
  }
]
```

```ts
// server/ssr.ts（示意）
import { renderToString } from 'react-dom/server'
import { routes } from './router'
import App from '../src/App'

export async function handleRequest(url: string) {
  const route = routes.find(r => r.match(url))
  const loaded = route ? await route.load({ url, headers: {} }) : { data: null }

  const html = renderToString(<App />)
  // 把数据注入到 HTML 里，客户端 hydration 时复用，避免重复请求
  const state = JSON.stringify(loaded.data).replace(/</g, '\\u003c')
  return `<!doctype html>
<html>
  <head></head>
  <body>
    <div id="root">${html}</div>
    <script>window.__SSR_DATA__=${state}</script>
  </body>
</html>`
}
```

关键点不是代码长什么样，而是“数据来源是可定位的”，从而缓存、鉴权、灰度都能落到同一个入口。

## Dev 与 Build 的差异：SSR 下差异更危险，因为它会变成线上事故

在 CSR 项目里，dev 与 build 的差异很多时候只是“性能差异”。在 SSR 项目里，差异更危险，因为它会演变成：

- dev 能跑，build 后服务端启动失败（依赖打包方式不同）。
- dev hydration 正常，线上 hydration mismatch（代码分支不同、数据注入不同、序列化差异）。
- dev 只有单进程，线上多进程/多实例导致缓存不一致（在内存里缓存了不该缓存的东西）。

Vite 的 dev 侧更接近“加载源码并转换执行”，build 侧更接近“打包产物并执行”。你必须把“差异可回归”作为工程目标：至少要有一个最小 SSR 产物 smoke，确保 build 产物能启动并渲染关键路由。

如果你的团队没有这个约束，SSR 会非常脆：每次升级依赖、升级 Vite、改插件都会带来不可预测的线上风险，最后大家会把 SSR 视为“难以维护的黑科技”，这不是 SSR 的问题，是工程纪律的问题。

## 混合渲染的正确姿势：先确定成本函数，再确定页面策略

混合渲染最怕“随心所欲”。我建议你用成本函数驱动策略，而不是用技术偏好驱动策略。一个可用的成本函数至少包括：

1. 首屏价值：首屏快对业务是否关键？（内容站通常关键）
2. 个性化程度：页面是否强依赖用户态？（强依赖意味着 SSR 缓存复杂或收益下降）
3. 更新频率：内容是否稳定？（稳定意味着 SSG/长缓存收益大）
4. 交互密度：交互是否重？（重交互往往客户端更合适）
5. 运行成本：你能接受多少服务端 CPU/内存/带宽开销？

有了成本函数，策略选择会变得自然：

- 内容稳定且首屏价值高：SSG 或 SSR + 长缓存。
- 内容变化快但可缓存：SSR + 短 TTL + 后台刷新。
- 强个性化：CSR 或 SSR 仅渲染壳，核心数据客户端请求。
- 重组件（3D、可视化）：客户端延迟加载，SSR 输出占位。

混合渲染的工程目标不是“每个页面都最优”，而是“整体成本可控、行为可解释、策略可演进”。

## 缓存是 SSR 的第二语言：没有缓存策略的 SSR 只是在烧钱

SSR 的成本主要来自渲染与数据获取。你如果不做缓存，SSR 很快会变成“把 CPU 当带宽用”的系统：每个请求都做同样的渲染、同样的数据查询。即便你的页面渲染很快，数据层也会成为瓶颈。

我建议把缓存分成三层治理：

1. 数据缓存：接口响应或聚合数据的缓存（更细粒度，复用更强）。
2. 片段缓存：某些可复用片段（例如导航、页脚、部分组件树）的缓存。
3. 页面缓存：整页 HTML 的缓存（收益最大，但失效最难）。

很多团队一上来就做整页缓存，结果发现难以处理用户态、A/B、权限差异，最后回退。更稳妥的路径往往是从数据缓存开始：它不依赖 HTML，一致性更容易保证；当你把数据缓存稳定下来，再考虑页面缓存就更自然。

缓存还有一个经常被忽视的点：失效策略必须可解释。尤其是混合渲染场景，你很容易出现“列表是新数据，详情是旧数据”这类体验割裂。解决方案不是“全部缩短 TTL”，而是把数据与页面的依赖关系写清楚，确保关键链路的失效边界一致。

## 常见误区/反模式：SSR 失败往往不是渲染失败，而是边界失败

常见误区/反模式我建议重点警惕：

1. 把 SSR 当成 SEO 开关：只要能输出 HTML 就行，不管数据、缓存与一致性。结果是首屏看起来快，但线上成本爆炸且经常 mismatch。
2. 在组件里随意请求数据：服务端请求一次，客户端 hydration 后再请求一次，导致重复开销与闪烁。
3. 同构代码里偷用浏览器 API：通过 `typeof window !== 'undefined'` 到处打补丁，最后 shared 代码不可测试、不可推理。
4. 用“内存单例缓存”当全局缓存：本地 dev 没问题，上线多实例后缓存命中率下降且行为不一致，甚至缓存了用户态导致数据串号。
5. 只在 dev 验证 SSR：不做 build 产物验证，导致上线才发现某个依赖打包后无法在 Node 运行。

如果你只能改一条，我建议改“组件内请求数据”。它是 SSR 体系化失败的根源之一：它把数据层从可治理的入口分散到组件树里，缓存与一致性都会随之崩坏。

## 我的判断：SSR 成功的关键不是“渲染在服务端”，而是“边界可治理”

我的判断是：SSR 的价值来自“把首屏与索引能力变成系统能力”，但它的成本来自“把运行时边界与数据边界强行暴露”。Vite 给了你非常强的灵活性，但灵活性不是免费的：它要求你用工程组织来换取确定性。

如果你要在团队里长期维护 SSR/混合渲染，我建议你优先做三件事：

1. 定义并守住目录边界：shared/server/client，让违规变得显而易见。
2. 把数据获取从组件树中抽离到路由单元（或等价层），让缓存与鉴权有入口。
3. 建立 build 产物的最小回归：至少能启动、能渲染关键路由、能通过 hydration 的一致性检查。

做到这三点，API 选择反而变成了细节：你可以用 Vite 的 SSR 能力，也可以迁移到更框架化的方案，工程的骨架不会变。反过来，如果骨架不存在，任何 SSR 方案都会在复杂度面前迅速失控。
