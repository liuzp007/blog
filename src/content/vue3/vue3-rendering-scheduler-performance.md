很多“Vue 3 性能优化”文章会从 API 清单开始：`computed`、`watch`、`v-memo`、`shallowRef`、`markRaw`、`keep-alive`。这些工具当然有用，但如果你把它们当成“提速按钮”，最终会陷入两种局面：

1. 你加了很多优化点，代码复杂度显著上升，性能却没有本质改善。
2. 你偶尔提速了，但一旦业务继续增长，性能又退化，而且你很难解释“为什么会退化”。

原因在于：Vue 3 的核心性能优势不是某个 API，而是它用“依赖关系 + 调度队列”管理更新。你要优化，就应该从“依赖关系是否真实”“更新是否可合并”“作业是否可去重”入手，而不是从“哪里可以缓存”入手。

本文尝试给你一个工程可用的解释框架：当页面卡顿时，你应该先问哪些问题、看哪些信号、修改哪些结构；以及为什么这些修改在 Vue 3 的调度模型里能稳定产生收益。

## 渲染不是“执行 render”，而是“执行一个带调度语义的 effect”

在 Vue 3 里，组件渲染可以视为一个响应式 effect：它会读取响应式状态，形成依赖集合；当依赖变化时，它不会立刻同步重渲染，而是进入调度队列。

这句话的重要性在于：你在组件内写的每一次响应式读取，都可能成为“未来触发更新”的原因。也就是说，性能问题本质上是依赖关系问题：

1. 依赖过多：一个渲染 effect 订阅了太多状态，导致任何细小变化都触发大范围更新。
2. 依赖不稳定：每次渲染读取的依赖集合变化很大（分支读取、动态 key），导致触发行为不可预测。
3. 依赖失真：渲染依赖了一些本不该影响 UI 的状态（例如统计、缓存中间态、临时计算结果），导致无意义更新。

如果你只在“更新很频繁”这个层面做优化，往往会错过根因：更新频繁是结果，依赖建模才是原因。

## job queue：把更新当成“作业”，收益来自去重与合并

Vue 3 的调度器可以概括为两步：

1. `trigger` 产生“这个 effect 需要重新跑”的意图。
2. 调度器把 effect 包装成 job 放入队列，做去重，然后在合适时机 flush。

你可以用下面的伪代码理解核心语义（依然是心智模型，不是源码）：

```ts
type Job = () => void
const queue: Job[] = []
const queued = new Set<Job>()
let flushing = false

function queueJob(job: Job) {
  if (queued.has(job)) return
  queued.add(job)
  queue.push(job)
  if (!flushing) {
    flushing = true
    Promise.resolve().then(flushJobs)
  }
}

function flushJobs() {
  try {
    for (const job of queue) job()
  } finally {
    queue.length = 0
    queued.clear()
    flushing = false
  }
}
```

工程意义非常直接：

1. 去重：同一组件在一个 tick 内被多个依赖触发多次，最终只渲染一次。
2. 合并：多次状态变更合并成一次渲染，避免中间态带来的无意义重算与 DOM patch。
3. 顺序：flush 队列可以排序，保证父子组件更新、watch 回调、post-flush 任务按语义执行。

当你理解了这些收益来源，你会知道该怎么优化：让 job 更少、更稳定、更可合并，而不是让某个更新“更快执行”。

## flush 语义：性能问题经常是“时序语义错误”伪装出来的

在业务里经常出现一种现象：你为了“让 UI 及时更新”，在 watch 里写了大量同步逻辑，或者在事件处理里连续 set 多个状态，然后为了修正时序再 `nextTick`。这类代码的典型问题是：你在和调度器对抗。

Vue 3 的 flush 语义（宏观上）可以理解为三类：

1. pre：DOM 更新前执行（适合准备工作、同步派生，但要谨慎，容易影响渲染链路）。
2. render：组件渲染与 patch（核心链路）。
3. post：DOM 更新后执行（适合读取布局、聚焦、动画启动等）。

当你把“应该 post 执行的逻辑”放进 pre 或同步执行，就会出现两种后果：

1. 性能后果：你把本可延后的工作挤进关键路径，扩大了帧时间。
2. 正确性后果：你读取了还没更新的 DOM 或旧的布局，导致补丁式修复（更多 nextTick），进一步恶化性能。

因此，很多性能问题的第一修复不是“减少计算”，而是“把工作移出关键路径”。这也是为什么调度语义是性能治理的一部分。

## 依赖关系才是“渲染范围”的决定因素

如果你把 Vue 3 的更新理解为“状态变了，组件就更新”，你会在优化上很被动。更准确的表达是：**依赖变了，订阅了该依赖的 effect 会被调度**。所以渲染范围不是由“组件层级”决定的，而是由“依赖图”决定的。

这带来一个工程策略：把高频变化的状态与大面积渲染解耦。

常见的解耦手段包括：

1. 把高频状态放在更小的组件/更小的 effect 范围里（例如拆分子组件、拆分渲染区域）。
2. 把高频状态从深层 reactive 换成 shallowRef，避免深层属性读取形成大量依赖。
3. 把不需要响应式的对象用 markRaw（或普通对象）隔离，避免被 track。

关键点在于：这些手段的本质都是“改变依赖图”，而不是“让某个点计算更快”。

## computed 与派生：把昂贵计算从“热路径”挪到“冷路径”

在 Vue 3 里，computed 的意义不只是缓存，而是把派生计算变成“按需求值”。当依赖变化时，computed 通常只标记 dirty，而不是立刻重算。重算发生在真正读取它的时候。

这对性能优化的启示是：你要把昂贵计算尽量表达为 computed，并让它的读取点尽量靠近真实消费点，而不是在很多无关地方提前读取。

同时要小心一个隐蔽坑：computed 返回的值如果是“新引用”（新对象/新数组），即便内容相同，也会导致下游依赖认为它变化了，从而触发额外更新。解决方案不是“禁用 computed”，而是：

1. 返回稳定引用（复用对象、复用数组、或拆分为多个原子 computed）。
2. 降低下游订阅面（下游只订阅需要的字段）。

## watch：让副作用有边界，否则你在制造额外 job

watch 的正确定位是副作用边界。你用 watch 做派生时，相当于自己维护一个调度系统：什么时候运行、运行顺序、去重、竞态、取消，都要你自己处理。结果就是：你引入了额外 job，且这些 job 可能挤进关键路径。

更工程化的做法是：

1. 派生用 computed：把依赖传播交给响应式系统。
2. 副作用用 watch：并把 flush 放在语义合适的位置（例如读布局放 post）。
3. 异步副作用要有取消语义：避免旧请求回写造成额外渲染与状态抖动。

下面是一个更接近工程现实的副作用写法（示意）：

```ts
import { ref, watch } from 'vue'

const q = ref('')
const data = ref<any[]>([])
const loading = ref(false)
const error = ref<string | null>(null)

watch(q, async (next, _, onCleanup) => {
  let cancelled = false
  onCleanup(() => { cancelled = true })

  loading.value = True
  error.value = null
  try {
    const res = await fetchData(next)
    if (cancelled) return
    data.value = res
  } catch (e: any) {
    if (cancelled) return
    error.value = String(e?.message || e)
  } finally {
    if (!cancelled) loading.value = false
  }
}, { flush: 'post' })
```

这里的关键不是 fetchData，而是取消语义：没有取消，旧请求回写会引发无意义更新；而且这种更新往往最难定位，因为它不是“频繁输入导致频繁请求”那么直观。

## 性能优化的正确问题：不是“哪里慢”，而是“为什么会触发”

当你观察到“渲染频繁”时，你应该优先问这四个问题：

1. 触发源是什么依赖？是哪个响应式 key 触发的？
2. 依赖范围为什么这么大？为什么这个渲染 effect 会订阅它？
3. 触发是否可合并？是否存在同 tick 多次触发同 job 的情况？
4. 触发是否必要？这个变化是否真的需要影响 UI？

这四个问题全部指向依赖关系与调度，而不是指向某段代码“执行慢”。很多时候，你并没有慢的计算，你只是做了太多次正确但不必要的计算。

## 常见误区与反模式：你以为在优化，其实在对抗调度器

1. 反模式：在一次用户交互里同步写很多状态，并在中途读取 DOM/布局，导致强制同步测量与多次 flush。正确做法是把 DOM 读取放到 post-flush。
2. 反模式：把大量派生逻辑写在 watchEffect 里，依赖集合随分支变化，导致更新范围时大时小，性能波动明显且难以复现。
3. 反模式：computed 返回新对象/新数组作为 props 传递，造成下游组件即便内容不变也被认为变了，触发额外渲染。
4. 误区：遇到卡顿就到处加 memo/cache。缓存只能降低单次执行成本，但更常见的问题是“执行次数过多”，你应该先减少触发次数与订阅面。
5. 误区：把 nextTick 当成万能胶水。nextTick 修正的是时序，不会修正错误的依赖建模；滥用会把问题从可解释的依赖图变成不可解释的“到处 await 之后才对”。

## 我的判断：Vue 3 的性能治理是“依赖图治理”，不是“局部技巧堆叠”

我认为 Vue 3 性能优化最重要的能力是解释能力：你能解释一个更新为什么发生、发生几次、在什么时机发生、能否合并、能否避免。能解释，才有稳定可复制的优化；不能解释，优化就会变成“运气好像变快了”。

因此我更推荐把性能工作当成“依赖图治理”：

1. 先让依赖真实：渲染只依赖渲染必要的状态。
2. 再让依赖稳定：避免动态依赖漂移与无意义的新引用。
3. 最后让调度可合并：把非关键路径移出 render 链路，减少 job 数量与 flush 压力。

当你按这个顺序做，Vue 3 的调度器就会成为你的盟友；反之，你会不断写出看似正确但不断对抗调度器的代码，最终把系统复杂度与性能成本一起拉高。
