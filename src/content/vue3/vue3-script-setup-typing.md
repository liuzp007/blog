很多团队在升级到 Vue 3 之后，会把“工程化体验变好”归因于一些表面变化：`<script setup>` 更短、Volar 更聪明、TS 报错更准确、模板里能补全。这样描述并不假，但它解释不了一个关键现象：为什么同样是 TypeScript，同样是组件，同样是模板，Vue 2/Options API 时代很难把类型体验做到这个水平，而 Vue 3 时代可以？

答案不在“写法更现代”，而在“信息流更干净”。`<script setup>` 把组件的公共接口从“运行时对象 + 约定”升级为“编译期可见的结构化信息”。当编译器能够看见你定义 props/emits/slots/expose 的方式，它就能把这些信息：

1. 投影到模板类型检查中，让模板不再是一个“黑箱字符串”。
2. 投影到组件使用方的类型中，让调用者得到可靠的 IDE 反馈。
3. 与重构动作绑定，使“改接口”变成可验证的工程操作，而不是祈祷式修改。

下面我会用工程视角说明：这次“质变”到底发生在哪里；你应该如何利用它降低系统复杂度；以及最容易被忽略、但会在中后期成本爆炸的反模式。

## 质变的根因：公共接口从“运行时约定”转为“编译期事实”

在 Options API 中，组件接口通常通过 `export default { props, emits, setup }` 这样的运行时对象表达。TypeScript 能看到对象字面量，但它很难把对象里“将来要被编译器消费的信息”与“运行时逻辑”做干净分层。更麻烦的是：

1. `props` 的类型表达既可能是构造函数数组，也可能是对象，也可能是 `PropType<T>` 的组合，推导路径复杂且容易退化为 `any` 或 `unknown`。
2. `emits` 的校验既可能是字符串数组，也可能是函数签名对象，不同写法会导致类型信息丢失或不一致。
3. 模板编译与 TS 类型检查之间缺少强连接：模板里出现了不存在的 prop 或错误的事件名，很多时候只能在运行时报错。

`<script setup>` 的核心变化是：组件的“接口定义”被写在顶层、以编译器宏的形式存在。宏的价值不在“魔法”，而在“它为编译器提供了稳定的 AST 锚点”。编译器拿到这些锚点后，可以把信息以一致的形式产出，供类型工具链消费。

你可以把这理解成一句工程语言：**接口声明被移到可被静态分析的区域，类型系统终于能持续地、低成本地参与“接口治理”。**

## defineProps：类型与默认值不再相互伤害

`defineProps` 的最佳用法不是“写得短”，而是“把类型约束当作接口的一部分”。你通常有两种路径：

1. 用 TS 类型参数表达 props 结构，让类型系统成为主来源。
2. 用运行时 props 选项表达校验与默认值，让运行时成为主来源。

在大型项目里，最容易出问题的点是：你希望“运行时有默认值与校验”，同时又希望“TS 能精确推导”。如果你把这两套东西分散在不同层，最终会出现类型与运行时不一致。

更稳妥的策略是：把“类型”作为接口事实，把“默认值”作为实现细节，通过编译器提供的桥接能力把二者对齐。

```ts
import { computed } from 'vue'

type Props = {
  id: string
  pageSize?: number
  mode?: 'light' | 'dark'
}

const props = withDefaults(defineProps<Props>(), {
  pageSize: 20,
  mode: 'light'
})

const isDark = computed(() => props.mode === 'dark')
```

这里的关键点不是 `withDefaults` 本身，而是：默认值与类型信息在同一个声明附近，变更成本极低。你在重构接口时，不需要同时去修改一堆离散的对象选项与类型定义。

## defineEmits：事件是 API，不是字符串

很多团队把事件当成“随便抛个字符串”。当项目变大，这会立刻变成治理问题：事件名冲突、事件参数不统一、调用方与实现方的契约漂移。

在 `<script setup>` 里，`defineEmits` 让你可以把事件当成“带签名的 API”。这件事非常关键，因为它把事件从“约定”变成“可验证契约”。你可以强制规定每个事件携带的 payload 结构，从而让跨组件通信更接近“函数调用”而不是“字符串拼接”。

```ts
type Emits = {
  (e: 'update:modelValue', value: string): void
  (e: 'submit', payload: { id: string; silent?: boolean }): void
  (e: 'error', err: Error): void
}

const emit = defineEmits<Emits>()

function onSubmit() {
  emit('submit', { id: '42', silent: false })
}
```

这段代码在工程上最大的收益是可重构性：你改事件名或 payload 字段，调用方会在编译期爆红，而不是在上线后某个边角交互里悄悄失效。

## 模板类型检查：让“模板”成为工程系统的一部分

Vue 的模板不是 TypeScript 文件，但它可以被“类型化”。这是质变的第二个关键点：当编译器能把 `<template>` 里用到的变量映射回 `<script setup>` 的声明环境，模板就不再是黑箱。

这会带来三个具体变化：

1. 你在模板里写错变量名、写错 props 字段、写错事件名，IDE 直接报错。
2. 你在模板里对 `ref`、`computed`、`props` 的使用能获得正确的类型提示。
3. 你可以把“组件 API”当作可搜索、可重构的符号，而不是一堆文本匹配。

注意：这不是“体验改良”，而是“质量机制”。模板类型检查本质上是把一类过去只能靠 code review 与测试兜底的错误，前移到编译期拦截。对于中大型团队，这是成本结构的变化：你用更低的边际成本换来更稳定的接口一致性。

## defineExpose：把组件当成对象时，请明确暴露面

任何框架都会遇到“不得不越过声明式边界”的时刻：例如你需要让父组件调用子组件的方法（聚焦输入框、滚动到某处、触发校验）。在 Vue 3 里，`defineExpose` 是一个很重要的治理点：它把“实例可访问性”从默认开放变成显式声明。

工程上你应该把它当作“最后的逃生门”，而不是常规通信手段。因为它会把组件间依赖从“数据流”转成“对象调用”，这通常意味着更强的耦合与更难的测试。

更好的使用方式是：

1. 只 expose 语义化动作，不 expose 内部状态。
2. 暴露的方法要稳定，避免暴露依赖内部实现的细节方法。

## 组件接口治理：props/emits/slots 是一套完整的协议

很多项目在定义组件时，只认真对待 props，而忽略 slots 与 emits 的契约。结果是：props 很强类型、slots 任意塞、emits 乱飞。长期看，组件系统会逐渐失去一致性。

在 Vue 3 的工具链下，你可以把三者作为“组件协议”的三条腿：

1. props：输入，描述组件需要什么。
2. emits：输出，描述组件能通知什么。
3. slots：结构扩展，描述组件允许哪里被替换、替换时能拿到什么上下文数据。

一个工程化的目标是：当你查看一个组件时，你能在顶层声明区看到完整协议，而不是在模板和逻辑里到处寻找隐含约束。这让组件更像“模块”而不是“片段”。

## 类型系统如何影响架构：把不变量写进签名，而不是写进注释

在大型项目里，真正昂贵的不是写代码，而是维护“不变量”：哪些值永远不为空、哪些状态不会同时出现、哪些事件必须按顺序触发。过去你可能用注释、文档、口头约定维护这些不变量；现在你可以把其中一部分写进类型签名，让工具链替你做持续校验。

例如，一个列表组件的状态常见不变量是：`loading` 为 true 时不应该同时有 `error`；`error` 存在时数据可能是旧的但不应被当成新鲜数据使用。你可以用更精确的联合类型表达状态机，让调用方按规则处理。

这不是“类型洁癖”，而是把系统复杂度从“运行时排查”转移到“编译期约束”。当团队规模增长，这种转移通常是净收益。

## 常见误区与反模式：类型体验变好之后，更容易写出隐蔽耦合

1. 反模式：把所有东西都塞进 `defineProps<{...}>()`，然后在组件内部随意修改入参衍生状态。props 是输入，应该尽量保持只读语义，派生状态应由 computed 或内部 state 承担。
2. 反模式：为了“省事”把事件参数写成 `any` 或 `unknown`，最终把错误从模板前移又推回运行时。事件签名是跨组件契约，越是公共组件越不应该模糊。
3. 反模式：大量使用 `defineExpose` 进行跨层调用，形成“组件实例依赖网”。短期开发快，长期重构成本极高，因为你把依赖从数据流变成了对象图。
4. 误区：把 `<script setup>` 当作“没有结构的自由区”，所有逻辑平铺在顶层。写法自由不代表结构自由，你仍然需要按能力域拆分 composable，把组件文件控制在可理解范围。
5. 误区：把模板类型检查当成“IDE 小甜头”。它是质量机制，不应被随意关闭或忽略，否则你会回到 Vue 2 时代那种“模板错了靠测试兜”的成本结构。

## 我的判断：script setup 的价值在“接口可治理”，不是“代码更短”

我更看重 `<script setup>` 带来的治理能力：它让组件接口变成编译期事实，进而让类型系统参与到接口的演进中。对团队来说，这意味着两件事：

1. 组件库可以更像“产品”：接口有版本感、有契约、有可验证的变更影响面。
2. 业务系统可以更像“系统”：能力抽象与组件协议更清晰，重构与迁移不再靠经验主义。

如果你的项目规模足够大，你会发现“少写样板”带来的收益很快耗尽；真正持续的收益来自“让错误更早暴露，让接口更难漂移”。这就是我认为 Vue 3 工程化体验发生质变的原因。
