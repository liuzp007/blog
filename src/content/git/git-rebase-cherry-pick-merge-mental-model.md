在很多团队里，rebase、merge、cherry-pick 的争论是一种“宗教战争”：有人坚持“rebase 一切”，有人坚持“merge 保真”，有人靠 cherry-pick 到处救火。争论的根源不是技术细节，而是缺少一个共同的心智模型：我们在用 Git 历史表达什么？我们希望历史为谁服务？我们愿意把成本放在哪里？

我给出一个更可执行的结论：这三种操作对应三种“语义承诺”。

1. Merge：承诺“保留并行演进的事实”，历史忠实反映协作过程。
2. Rebase：承诺“把一组变化重放到新的基线”，优先线性叙事与审阅体验。
3. Cherry-pick：承诺“抽取一段语义变化并搬运”，优先局部迁移与救火效率。

承诺不同，风险不同。你无法只要收益不要代价。

## 从 DAG 到叙事：Git 历史的本质不是“线”

Git 的对象模型是 DAG：提交指向父提交（可能多个父），分支只是指针。所谓“线性历史”只是你选择用某种方式展示 DAG 的结果，而不是 DAG 的本体。理解这一点很关键：rebase 不会“改变事实”，它是“重新编码事实的表达”；merge 也不是“更真实”，它只是选择保留并行节点的显式结构。

把历史当作协作界面时，你要问：这个界面要表达“并行发生过”，还是表达“最终形成的因果链”？两者都合理，但适用于不同场景。

## Merge：保真与可追溯，但审阅成本可能更高

merge 的核心价值在于保留并行。它让你在历史里看到“两个分支曾经并行存在”，并且用一个 merge commit 表达“它们在某个点汇合”。这对以下场景很有价值：

1. 需要追溯协作过程：什么时候分叉、什么时候汇合、冲突在哪里被解决。
2. 需要明确边界：一个 feature 分支的生命周期在 merge commit 上很清晰。
3. 需要稳定的回滚：如果你的团队用“merge commit 作为发布边界”，回滚可以围绕边界做。

但是 merge 的代价是：如果你把分支上的 WIP 提交、噪音提交、修修补补提交原样带入主线，主线会迅速变得不可读。所谓“merge 保真”，如果保留的是噪音，那只是把成本转移给评审者和未来维护者。

## Rebase：审阅友好与叙事清晰，但会改写身份与时间线

rebase 的本质是“重放”：把一串提交在新的基线上重新生成一串新的提交。新的提交有新的 SHA，旧 SHA 不再是同一对象。这带来两个重要后果：

1. 身份改写：任何引用旧提交 SHA 的东西（评审链接、故障定位记录、聊天引用）都会失效或语义漂移。
2. 时间线重编码：提交时间与父子关系变化，`git blame` 的指向可能更符合“最终叙事”，也可能更模糊“真实发生过的过程”。

因此我对 rebase 的使用边界很明确：

1. 在未被他人依赖/发布的分支上 rebase：可以，成本可控。
2. 在已经被共享的分支上强行 rebase：通常不值得，除非你明确承担同步成本，并且团队有成熟流程。

一个常见且健康的模式是：个人分支随便 rebase，合并到主线时用一种明确策略（比如 squash 或 rebase-merge），但主线本身尽量少做历史改写。

## Cherry-pick：局部迁移的快刀，但会制造“平行宇宙”

cherry-pick 很像把某个提交当作“补丁”搬运到另一个分支。它的优势在于快，尤其是在以下场景：

1. 线上 hotfix：主线还在向前推进，但你需要把一个修复搬到 release 分支。
2. 多版本维护：比如同时维护 v2/v3，你需要把修复同步到多个分支。
3. 回滚后的再引入：某功能被回滚，但其中某个子修复仍然需要保留。

代价是“平行宇宙”：同一语义变化会在不同分支有不同的提交对象（不同 SHA），未来你想合并这些分支时，Git 无法自动识别“它们实际上做了同一件事”，冲突会更频繁，历史也更难解释。

因此 cherry-pick 的边界是：把它当作“救火工具”而不是“日常同步方式”。如果你发现团队日常靠 cherry-pick 同步功能，很可能说明分支策略或发布策略有结构性问题。

## 选择标准：你在优化哪一种成本

把三者放在一张表里看，会更清楚：

1. Merge 优化：协作过程可追溯；代价是审阅噪音与历史复杂度。
2. Rebase 优化：审阅与叙事清晰；代价是身份改写与共享分支同步成本。
3. Cherry-pick 优化：局部迁移速度；代价是平行宇宙与后续合并复杂度。

所以选择不是“哪种更高级”，而是“我们当前最痛的成本是什么”。在一个高频发布、强审阅、强回滚要求的团队里，你可能更希望主线历史清晰，并为回滚设计边界；在一个需要强保真审计的环境里，你可能更强调 merge 的过程可追溯性。

## 实战：在同一个团队里组合三者，而不是二选一

现实里很少是“全 rebase”或“全 merge”。更合理的是定义组合：

1. 个人分支：允许 rebase（甚至鼓励），把 WIP 整理成可审阅序列。
2. 主线合并：使用 merge commit 或 squash merge（取决于你要不要保留并行事实）。
3. 发布分支：谨慎 cherry-pick 修复，但记录清楚“从哪里摘来的”。

下面是一个典型流程示例：开发分支上重放并整理，合并到主线用 merge commit 保留边界：

```bash
# 1) 在 feature 分支上整理提交（仅限未发布/未被他人依赖的分支）
git fetch origin
git rebase -i origin/main

# 2) 合并到 main，用 merge commit 保留“这是一次集成点”
git checkout main
git merge --no-ff feature/xxx -m "merge: feature xxx"
```

如果你希望主线更线性，又希望保留分支结构，可考虑 rebase 的 `--rebase-merges`（但要清楚它更复杂，且更容易踩坑）：

```bash
git fetch origin
git rebase --rebase-merges origin/main
```

## 冲突处理的真实含义：你把冲突放到哪里，就把成本放到哪里

很多人以为“rebase 冲突更多”。更准确的说法是：rebase 把冲突分摊到每个提交的重放过程；merge 把冲突集中到一次合并点。两者的总冲突量取决于变化的交错程度，不是操作名称。

分摊与集中对应不同的风险：

1. 分摊（rebase）：你可能在某个提交上解决冲突，但语义上应该在另一个提交解决。历史的“因果链”会被污染。
2. 集中（merge）：一次冲突解决可能混入多种语义变化，回滚和 blame 会更难。

我的经验是：如果你能保证提交序列本身是“可验证的叙事”，分摊冲突是有利的，因为它让每一步保持一致性；但如果你的提交本来就是 WIP，分摊只是把混乱拉长。

## 常见误区：把操作当成风格，把代价当成别人的问题

这里列几个非常典型的反模式：

1. 反模式：共享分支上强制 rebase，然后要求所有人“自己处理一下”。
   结果：同步成本爆炸，review 链接失效，团队信任下降。
2. 反模式：把 merge 当垃圾桶，什么提交都往主线倒。
   结果：主线不可读，任何 blame 都像抽奖。
3. 反模式：大量 cherry-pick 来回同步 feature。
   结果：平行宇宙扩大，最终合并时冲突成倍增长。
4. 反模式：为追求“线性”，把真实发生过的并行事实全部抹掉。
   结果：历史看起来漂亮，但复盘与追责丢信息。

这些误区背后的共同点是：把历史当作个人偏好，而不是团队协作界面。

## range-diff：重写历史后，如何证明“语义没变”

当你 rebase 或者做了大规模历史整理后，最怕的是“看起来一样，但其实变了”。`git range-diff` 是一个很实用的工具：它比较两段提交序列的差异（以提交为单位，而不是文件 diff），让你看到“哪些提交被改了、顺序怎么变了、内容怎么变了”。

示例：

```bash
# 比较 rebase 前后两段提交序列
git range-diff origin/main...HEAD@{1} origin/main...HEAD
```

你不需要把它当成日常命令，但在“历史整形”这种高风险操作之后，它是我认为最值得使用的验证手段之一。

## 我的判断：不要争“统一操作”，要统一“历史语义”

我不建议团队规定“必须 rebase”或“必须 merge”。我建议团队统一的是“历史语义”：主线历史希望表达什么？合并点是否需要边界？事故时回滚依赖哪种结构？评审者希望看到怎样的叙事？

当语义统一后，操作会自然收敛为少数几种组合策略。反之，强推某个操作只会把成本转移给最倒霉的人：通常是评审者、维护者、值班工程师，以及半年后的你。
