线上事故发生时，工程团队往往会同时失去两样东西：时间与确定性。时间很好理解，确定性则更隐蔽：你不确定“问题从哪里开始”，不确定“什么状态算安全”，不确定“回滚会不会引入更大的事故”。很多团队把事故处理理解成“赶紧改代码”，但真正决定止损速度的，往往是你能不能在 5 分钟内回答两个问题：

1. 现在代码处于什么状态？它是怎么到这里的？
2. 我们能以最小风险回到哪个状态？这个状态为什么安全？

我对 Git 在事故中的定位很明确：它不是“存代码的地方”，它是你在不确定性里重建事实的工具链。Reflog 用来找回“你以为丢了”的提交；Bisect 用来把“怀疑很多地方”变成“一个可验证的嫌疑点”。两者共同服务一个目标：最小恢复路径（Minimal Recovery Path）。

这篇文章不是工具说明书。我不会给你罗列命令选项。我只讲一套流程：从事故现场到恢复点的路径如何设计，如何避免把事故处理变成第二次事故。

## 事故现场的第一原则：先止血，再取证，再修复

很多事故升级的原因不是“技术不够”，而是“顺序错了”。正确顺序是：

1. 止血：先恢复服务到一个可接受的状态。这个状态可能不是最优，但要稳定、可回滚、可解释。
2. 取证：保留证据（当前 commit、构建号、配置快照、日志、回滚动作记录）。取证不是给甩锅，是为了防止重复踩坑。
3. 修复：在隔离环境里定位根因，做最小修复，验证后再推。

Git 在这三个阶段都能提供价值，但它的典型误用是：团队一上来就“改 main、强推、回滚又前滚”，把历史和证据一并抹掉。这样你就失去了复盘能力，也失去了下一次事故的止损能力。

把“事故处理”当作一套可复用的流程，而不是临场发挥，是成熟团队的分水岭。

## Reflog 是什么：它记录的是“引用的移动”，不是“提交的存在”

很多人把 reflog 当成“神奇的回收站”，以为它记录了所有提交。更准确的理解是：reflog 记录的是引用（ref）在本地如何移动过，比如 `HEAD`、分支指针、`stash` 指针等。提交对象本身存在于对象数据库里，只要没有被垃圾回收（GC）清理，理论上就能找回；reflog 提供的是“我该从哪里找”的路径。

这带来三个关键结论：

1. reflog 主要是本地证据。远端是否保留、保留多久、谁能看到，取决于服务端实现与策略，你不能假设远端一定能救你。
2. reflog 的价值在于“把误操作可逆”。reset、rebase、checkout、merge、pull 这些会移动引用的操作，都能在 reflog 里留下轨迹。
3. reflog 不是无限期保险。它会过期，会被 GC 影响。事故处理要把“找回”变成及时动作，而不是事后祈祷。

你要把 reflog 看成黑匣子：它不保证你永远能还原一切，但它能在关键窗口期给你足够的事实。

## 最小恢复路径：把“回滚”拆成可验证的三个动作

我把最小恢复路径定义为三个动作的组合，每个动作都要可验证：

1. 定位“安全基线”：明确一个 commit 或 tag，作为“已知可用”的状态。
2. 回到安全基线：用最小风险的方式把生产带回这个状态。
3. 固化恢复点：让团队与系统都知道“我们回到了哪里”，并能在必要时继续前进。

这里的关键是“可验证”。一个没有验证的回滚只是心理安慰。

在 Git 层面，最常见的“安全基线”来源包括：

1. 上一次成功发布对应的 tag（最推荐）。
2. 主分支上某个已知稳定的 merge commit（次推荐）。
3. CI 产物的版本号对应的 commit（如果你有强绑定）。

当你有了安全基线，回滚方式取决于你的发布模型。纯 Git 层面的回滚通常是：

1. `git revert`：保留历史，新增一个反向提交，适合主线协作与审计。
2. `git reset --hard` + 强推：改写历史，风险大，只适合非常明确且受控的场景（例如临时分支、误推到错误分支且尚未被他人依赖）。

事故处理中，默认应该偏向 `revert`，因为它把风险显式化：你不会在回滚过程中抹掉证据。

## 用 Reflog 找回“丢失的提交”：从 panic 到可控的具体步骤

最常见的事故不是线上 bug，而是“误操作把正确的东西弄丢了”，例如：

1. `git reset --hard` 回错了提交。
2. `git rebase` 做坏了，冲突解决后语义被污染。
3. 强推覆盖了远端分支，大家都拉到了错误状态。
4. 误删分支，以为提交没了。

当你意识到“我把东西弄丢了”，先不要做更多操作。第一步是把现场保存成一个新分支，避免你继续移动 `HEAD` 导致证据被覆盖：

```bash
# 1) 先把当前状态固定下来，避免继续操作污染现场
git status
git branch rescue/incident-20260324
```

然后看 reflog。最常用的入口是 `HEAD` 的 reflog：

```bash
# 查看 HEAD 的移动历史
git reflog --date=iso

# 常见用法：只看最近 30 条
git reflog -n 30
```

你会看到类似 `HEAD@{5}` 这样的条目。核心不是记住格式，而是找到“那次误操作之前”的位置。找到后，你有两种策略：

1. 创建一个新分支指向那个位置（最安全）。
2. 直接 reset 回去（更激进）。

我推荐先分支，再决定怎么处理：

```bash
# 假设你要找回的点是 HEAD@{12}
git branch recovered/head-before-mistake HEAD@{12}

# 然后你可以 checkout 过去检查
git checkout recovered/head-before-mistake
```

如果你要恢复的是某个分支指针的移动，也可以看对应分支的 reflog：

```bash
git reflog show main -n 30
git reflog show feature/xxx -n 30
```

关键思路是：不要急着“把主线改回去”，先把正确的对象找回来并固定住。只要你把它固定成分支或 tag，它就不会轻易被 GC 清理，你就重新拥有了确定性。

## Bisect 的价值：把“猜”变成“可重复的定位过程”

很多团队的定位方式是“翻 diff + 翻群聊 + 猜”。在简单项目里凑合能用，但在多人协作和高频发布下，这会变成时间黑洞：你会不断做无效尝试，并且每次尝试都在消耗团队注意力。

Bisect 的核心价值是分治：你给它一个“坏”的点和一个“好”的点，它用二分法逼近第一个引入问题的提交。它把定位从“我觉得是这个”变成“我能证明是这个”。

但要注意：bisect 的前提是“可判定”。你必须能对任意中间状态给出一个可重复的判断：好或坏。这个判断可以是自动化测试、编译是否通过、某个关键接口是否返回正确值、关键页面是否能渲染等。

一个典型流程如下：

```bash
# 1) 找到一个确定坏的点（通常是当前 HEAD）
git bisect start
git bisect bad

# 2) 指定一个确定好的点（例如上一次稳定发布的 tag）
git bisect good v1.2.3

# 3) Git 会自动 checkout 到中间点，你在该点判断好/坏
# 判断后标记：
git bisect good
# 或
git bisect bad

# 4) 重复直到定位到第一个坏提交
git bisect reset
```

真正的难点在第 3 步：你怎么判断“好/坏”既可重复又足够便宜？如果每次判断都要半小时，那二分法也救不了你。

## 把 Bisect 变成工程能力：用脚本固化“好/坏判定”

成熟团队会把 bisect 的判定写成脚本，这样定位过程可以被复用、被并行、被复盘。`git bisect run` 可以让你把“判定逻辑”变成一个命令：返回 0 表示 good，返回非 0 表示 bad。

示例：用单元测试作为判定（只是示意，命令按你的项目调整）：

```bash
git bisect start
git bisect bad
git bisect good v1.2.3

# 用脚本自动跑判定：0=good, 1=bad
git bisect run bash -lc 'pnpm test --filter critical && pnpm build'
```

这类脚本的价值不在于“更自动化”，而在于“更可解释”：当你得到一个坏提交时，你知道它是在哪个判定规则下被判坏的。这对复盘和修复策略选择非常重要。

我建议你在事故结束后，把这类脚本收敛为“回归测试的最小集合”：下一次同类事故，你就能更快复用判定逻辑，而不是重新发明。

## Reflog + Bisect 的组合拳：先恢复确定性，再定位根因

两者的关系是这样的：

1. Reflog 用于恢复确定性：找回正确的提交/状态，避免你在错误的地基上定位。
2. Bisect 用于缩小不确定性：在“好与坏”之间找出第一个引入问题的变化。

一个高频事故场景是：你本来在做 rebase 或者修复，结果把历史弄乱了，随后线上出了问题。你此时如果直接 bisect，很可能是在一个被污染的历史上二分，最后定位到一个“看起来相关但其实只是 rebase 冲突解决造成的副作用”的提交。

正确做法通常是：

1. 用 reflog 找回“未污染”的基线（例如 rebase 前的 HEAD、发布 tag 对应的 commit）。
2. 在该基线上标记 good/bad，再 bisect。

这样你定位到的是“真实引入问题的语义变化”，而不是“你在事故中制造的噪音”。

## 常见误区：事故处理把 Git 当作武器，而不是安全带

下面这些反模式我见过太多次，它们的共同结果是：事故处理变成更大的事故。

1. 反模式：在主分支上直接 `reset --hard` + 强推做回滚。
   结果：证据被抹掉，其他人本地状态失配，后续修复与复盘成本暴涨。
2. 反模式：没有固定现场分支就开始一顿操作。
   结果：reflog 条目被新的移动淹没，你丢掉了最关键的恢复线索。
3. 反模式：把 bisect 当成“最后手段”，前面先猜三小时。
   结果：浪费注意力窗口，错过最有效的止损时间。
4. 反模式：bisect 判定不稳定（受网络、缓存、外部依赖影响），导致结果不可复现。
   结果：定位到的提交无法证明，团队进入“再次猜”。
5. 反模式：把回滚等同于“恢复”，忽略配置、数据、缓存的版本兼容。
   结果：代码回滚了，系统仍然坏；大家以为 Git 不可靠，实际是恢复路径不完整。

事故里你最需要的是确定性，而强推、频繁改写历史、缺少证据固定，都是在主动制造不确定性。

## 我的判断：把恢复路径产品化，Git 才会在事故中真正救你

reflog 和 bisect 都不是“高手命令”。它们只是把事故处理从“临场发挥”变成“可复用流程”的两个抓手。我的判断是：你不应该把它们当作个人技巧，而要把它们产品化到团队流程里：

1. 发布必须产出可引用的“安全基线”（tag 或可追溯版本号）。
2. 每次事故结束都要沉淀最小判定脚本，让下一次 bisect 更快。
3. 主线回滚默认用可审计的方式（revert），除非有非常明确的例外条件与授权流程。
4. 任何高风险操作前先固定现场分支，把确定性先留住。

当你把这些当作团队接口，而不是个人偏好时，事故处理的速度和质量会发生质变：你会更少依赖“英雄主义”，更多依赖“流程的可重复”。而这正是成熟工程组织在不确定性里保持稳定产出的关键。
