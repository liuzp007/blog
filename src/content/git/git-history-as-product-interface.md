很多团队把“提交历史”当作副产物：能跑就行，能合就行。等到线上事故、回归地狱、评审疲劳、交接失忆一起出现时，才想起历史的价值。我的观点是：在资深团队里，Git 历史不是日志，而是产品界面。它有用户（评审者、排障者、接手者、合并者、你未来的自己），有使用场景（追责、回滚、定位、复盘、迁移、合规），也有明确的“接口契约”（读起来不痛、改起来不炸、回滚有路径）。

这篇文章不讲“怎么写 commit message 的格式”这种入门清单。我只讨论一件事：把历史当作界面时，提交应该如何被设计，才能把协作成本从“爆炸”降回“可控”。

## 历史的用户是谁：你在为谁写提交

一个提交序列至少面对四类用户：

1. 评审者：他不是在读代码，他是在读“变化的理由”。评审需要“可解释的分割”，否则只能靠直觉打补丁式评论。
2. 排障者：他需要把问题归因到一个最小集合的变化，并且快速做对的回滚或前滚决策。
3. 维护者/接手者：他要在没有上下文的情况下理解“为什么这样写”，而不是只看见“这样写了”。
4. 自动化系统：CI、发布、回滚、changelog 生成、依赖升级机器人都在消费历史。它们不理解你的语气，但理解你的结构。

把历史当产品界面，意味着你要明确：每一个提交对这些用户提供了什么信息，制造了什么成本。

## 历史的“接口契约”：可读、可回滚、可追责、可组合

我用四条契约来评估提交质量（不是 message，而是“提交粒度与结构”）：

1. 可读：`git log -p` 能读出“发生了什么 + 为什么这么发生”，而不是一堆互相打架的 diff。
2. 可回滚：关键提交能用 `git revert` 或 `git reset` 组合出合理的恢复路径；不会把回滚变成一次新功能开发。
3. 可追责：`git blame` 指向的人能解释这行代码背后的意图，而不是“那次为了过 CI 的临时改动”。
4. 可组合：一组提交能被安全地迁移（cherry-pick）、重放（rebase）、或被其他分支复用，而不会把历史变成一次“手工冲突仪式”。

注意：契约之间有冲突。比如，把一个 PR 压成一个大提交有利于回滚（一步撤销），但可能损害可读与可追责（细节被抹平）。把 PR 拆得太细会提高可读，但可能降低可回滚（回滚需要多步且容易漏）。因此提交设计是权衡问题，而不是格式问题。

## “故事板”式提交：把变化组织成可验证的叙事

很多人误以为“提交越小越好”。我更倾向“提交是一个可验证的最小叙事单元”。所谓叙事，不是文学，而是评审和排障能用的逻辑顺序：

1. 引入约束或结构：让后续变化有落点。
2. 引入能力：实现核心逻辑。
3. 引入防护：测试、校验、监控、回退开关。
4. 引入收尾：文档、注释、清理、开关默认值。

一个具体例子：把一项“表格性能优化”做成可读的提交序列，通常比“同时改缓存、改渲染、改接口、改组件层级”更容易被评审。

下面是我常用的提交序列模板（示意，不是要求你照抄）：

```bash
git commit -m "perf(table): instrument slow paths"
git commit -m "perf(table): reduce render thrash by memoizing cell layout"
git commit -m "perf(table): add regression test for 5k rows"
git commit -m "perf(table): guard with feature flag and rollback plan"
```

注意第一个提交“先测量再优化”。这不是哲学，是为了让历史具备可追责与可复盘性：当优化引入回归时，你能快速知道是“策略错了”还是“实现错了”。

## 提交粒度的核心指标：冲突半径与回滚半径

我不用“改了多少行”来决定是否拆提交，我用两个半径：

1. 冲突半径：这个提交在未来被重放/合并时，会与多少并行变化发生冲突？
2. 回滚半径：这个提交如果要撤销，会影响多少无关功能或无关模块？

一个“改配置 + 改业务逻辑 + 改 UI + 改测试”的提交，冲突半径通常很大，因为配置与业务逻辑在不同时间线会被不同人改；回滚半径也很大，因为撤销它会把测试也撤销，或者把 UI 也撤销。

更合理的拆法是把“边界稳定、冲突风险低”的部分单独成提交。比如“纯重构但不改行为”的变更，通常可以单独放在一个提交里，因为它有利于后续行为变更的 diff 可读性，也有利于排障时判断问题是否来自行为变化还是结构变化。

但要警惕：所谓“纯重构”必须可验证。否则它只是你对自己的安慰。

## 评审友好的提交：让 diff 结构与审阅策略对齐

你无法控制评审者的注意力，你只能降低注意力的浪费。一个评审友好的提交应当把“最难审的东西”隔离开：

1. 机械性改动（lint、format、rename）与语义改动分离。
2. 迁移性改动（比如 API 版本迁移）先做适配层，再逐步替换调用点。
3. 重构先落地“可回退的骨架”，再动核心路径。

常用的技巧之一是“先引入新实现并旁路，再切换调用点”。提交序列可以这样设计：

```bash
git commit -m "refactor(auth): add token provider abstraction"
git commit -m "feat(auth): implement v2 token provider behind flag"
git commit -m "chore(auth): switch default provider to v2"
git commit -m "chore(auth): remove v1 provider after bake time"
```

这样做的好处不是“更漂亮”，而是历史提供了可操作性：当 v2 出问题时，你可以 revert “switch default provider”，而不是在一坨提交里找开关。

## “可回滚”不是一句话：你要在历史里预留最小恢复路径

很多人说“可回滚”，实际是在 PR 描述里写一句“必要时可回滚”。真正的可回滚是：你能在事故发生后 5 分钟内执行一条命令，把系统带回一个安全状态，并且这个安全状态是可解释的。

我把恢复路径分成三类：

1. 单提交回滚：`git revert <sha>` 就能恢复。适合开关切换、配置变更、独立功能上线。
2. 线性序列回滚：需要 revert 一段范围。适合 feature 系列提交。
3. 版本回退/发布层回滚：依赖 CI/CD 或制品仓库，不完全依赖 Git。

如果你在设计提交时已经知道将来可能需要回滚某一类变化，就应该把它的边界设计为“可以用 revert 撤销”。这要求你在提交里避免混入无关修改，尤其是避免把“行为变化”和“清理改动”混在一起。

示例：回滚某个功能系列的安全方式通常是“按逆序 revert 一段提交”：

```bash
# 假设 A..B 是功能提交范围（A 更早，B 更晚）
git log --oneline A..B
git revert --no-commit B
git revert --no-commit <B-1>
git revert --no-commit <...>
git commit -m "revert: rollback feature X due to regression"
```

你当然也可以 `git revert A..B`，但你要理解：范围语法的边界和顺序，常常会被误用。事故时的目标是稳，而不是炫技。

## 常见误区：把“提交历史”当成“个人草稿箱”

下面这些是我在大多数团队里见过的高频反模式，它们几乎都会以“评审痛苦 + 回滚困难 + blame 失真”收场：

1. 反模式：一堆 `fix`, `tmp`, `wip`, `try again` 直接进主分支。
   结果：历史不可读，问题出现时无法定位“最终意图”。
2. 反模式：把格式化/重命名和功能改动混在一起。
   结果：diff 噪音淹没语义，评审只能靠运气。
3. 反模式：为了“一个提交可回滚”，把一个月的变化压成一个提交。
   结果：可回滚，但不可审、不可追责、不可复盘；回滚后你也不知道撤销了什么。
4. 反模式：用 rebase 把冲突解决过程写进历史，或者反过来，把真正的演进过程全部抹掉。
   结果：历史既不忠实，也不整洁；对任何用户都不友好。
5. 反模式：把依赖升级与业务改动混合。
   结果：回滚依赖升级会一起回滚业务，或者回滚业务导致依赖状态不一致。

我对“WIP 提交”的态度很明确：WIP 可以存在于分支、存在于你的本地、存在于草稿 PR，但不要把它作为产品界面交付到主线。你可以在合并前用 squash 或 rebase 把它整理为可读的历史，但不要把整理责任甩给未来。

## 工具与流程：把“历史设计”变成可执行的习惯

历史设计不是靠自律口号，靠的是流程与工具把成本外部化。这里给三条我认为最有效的实践：

1. 用 `fixup` 与 `autosquash` 把“评审修正”收敛到目标提交，而不是堆叠新提交。
2. 用 `range-diff` 在 rebase 后验证“语义是否被不小心改变”。
3. 用 merge queue/流水线把“历史整洁”和“可持续合并”结合起来，减少“为了合并而合并”的补丁提交。

示例：评审期间追加修正，正确方式通常是：

```bash
# 找到你要修正的那条提交
git log --oneline

# 以 fixup 方式提交（或 --squash）
git commit --fixup <target-sha>

# 自动把 fixup 归位并重写
git rebase -i --autosquash origin/main
```

这能让历史保持“每个提交都有完整叙事”，同时评审过程也不被破坏。

## 我的判断：历史的价值不在“整洁”，而在“可操作性”

把历史当作产品界面，最终目标不是“看起来干净”，而是“出事时能操作、协作时能复用、变更时能解释”。整洁只是一种手段：让信息密度上升，让噪音下降，让恢复路径变短。

如果你的团队还在争论“到底要不要 squash”，我建议换一个问题：我们希望历史服务谁？我们希望在事故发生时按哪个按钮？我们希望半年后 `git blame` 指向的人能回答什么？当你能回答这些问题时，提交设计会自然收敛为一套可执行的风格，而不是一份永远没人看的规范。
