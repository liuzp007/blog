## 先把结论放在前面：Vue 2 响应式是“可预测的妥协”

Vue 2 的响应式之所以在工程上成功，不是因为它无所不能，而是它在“开发者心智成本、运行时成本、实现复杂度、生态可兼容性”之间找到了一个长期稳定的平衡点：用 `Object.defineProperty` 把对象的既有属性转换为 getter/setter，把“读取”当作依赖收集入口，把“写入”当作触发更新入口；再用 Watcher 把依赖关系挂在组件渲染函数、计算属性和用户 watcher 上。

你会觉得它“好用”，是因为它把依赖关系构建成了默认行为：你不需要手动声明“我依赖什么”，只要在渲染/计算过程中读取过，依赖就自动形成。你会觉得它“会失手”，则是因为这套机制天然依赖“能拦截到读取/写入”，而 Vue 2 拦截不到三类关键动作：

1. 新增/删除属性（尤其是对象已有 key 集合之外的变更）。
2. 通过索引直接改数组元素长度或某个位置（尤其是 `arr[index] = x`、`arr.length = n`）。
3. 某些跨边界的可变对象（例如把同一对象同时交给 Vue 与非 Vue 逻辑管理，导致写入路径绕过 setter）。

工程上真正的难点不是“知道这些限制”，而是把它们翻译成可执行的约束：如何在团队里让这些限制不扩散成随机 bug，如何在架构里把不可拦截的变更收敛到少数出口。

## 机制复盘：依赖关系不是“监听数据”，而是“追踪读取”

很多人把 Vue 2 说成“监听数据变化”，这句话会把你带进错误的心智模型。Vue 2 实际追踪的是“谁读取了谁”。依赖关系建立的时机不是写入，而是读取；写入只负责通知。

一个更接近事实的抽象是：每个可响应式的属性都有一个 `Dep`，它是一个订阅中心；Watcher 在执行（渲染/计算/用户 watcher）时把自己设置为全局的 `Dep.target`，当 getter 触发时，getter 会把当前 watcher 订阅到对应的 dep 上。setter 则在值变化时 `dep.notify()`，把订阅者逐个唤醒。

下面这个伪码足以解释 80% 的行为（不追求源码一致，只追求机制准确）：

```js
class Dep {
  constructor() { this.subs = new Set() }
  depend() { if (Dep.target) this.subs.add(Dep.target) }
  notify() { for (const w of this.subs) w.update() }
}
Dep.target = null

function defineReactive(obj, key, val) {
  const dep = new Dep()
  let childOb = observe(val) // 深层递归
  Object.defineProperty(obj, key, {
    get() {
      dep.depend()
      if (childOb) childOb.dep.depend() // 让数组/对象整体也可被依赖
      return val
    },
    set(next) {
      if (next === val) return
      val = next
      childOb = observe(next)
      dep.notify()
    }
  })
}
```

注意：依赖关系是动态的、按运行路径建立的。也就是说，“某个组件是否会因为某个字段变化而更新”，取决于它上一次渲染是否读取了该字段。如果你的模板里有条件分支，依赖集合也会随分支改变而改变。这是响应式“聪明”的来源，也是某些诡异 bug 的根源。

## 设计边界一：对象新增属性为什么拦不住

Vue 2 的 defineProperty 方案决定了它只能改写“已有属性”的 getter/setter。新增属性是在对象上创建一个新的 key，这个 key 没有被 defineReactive 包装过，自然就没有 dep，也就没有 getter/setter 的依赖收集与 notify。

Vue 2 提供了 `Vue.set(obj, key, val)` 来弥补：它内部会对新 key 调用 defineReactive，并触发 obj 的“整体 dep”去通知使用者。团队里如果把 `Vue.set` 当成“语法细节”，那就是在把随机 bug 留给未来的自己。

关键的工程判断是：新增属性到底应该被允许存在吗？在大型项目里，我更倾向于把它当作一种“危险行为”，只在少数场景开放，并且用结构约束把它变得可控。

一个更工程化的策略：提前把对象的 shape 定死。也就是说，不要把“对象”当作自由扩展的字典，而是当作有 schema 的数据结构。你可以在初始化时把可能出现的字段都声明出来，即使初值是 `null` 或 `undefined`。

```js
data() {
  return {
    // 把 shape 固定下来，后续只做赋值，不做新增 key
    form: {
      name: '',
      age: null,
      address: '',
      // 未来字段也预留
      extra: null
    }
  }
}
```

这样做的收益是“双赢”的：运行时层面减少漏更风险；协作层面降低联调不确定性；代码阅读层面让数据结构更可推断。

## 设计边界二：数组索引与 length 为什么是特例

数组的索引是动态的、可扩展的，Vue 2 不可能为每一个潜在索引提前 defineProperty。于是 Vue 2 采取了另一条路：拦截数组的变异方法（push/pop/shift/unshift/splice/sort/reverse），在这些方法执行后对数组整体 dep 做 notify。

因此：`arr.splice(i, 1, x)` 能触发更新，`arr[i] = x` 不一定；`arr.push(x)` 能触发更新，`arr.length = 0` 不一定（至少不是通过 Vue 的拦截路径）。

工程上，这个边界常被误读为“数组更新有坑”。更准确的说法是：Vue 2 在数组上提供的是“约定型更新接口”，你用它认识的那几个方法，它就能保证可观察；你绕开它，它就不保证。这里没有道德判断，只有接口契约。

建议把“数组变更”统一收口成少数方法，尤其是在 store 或模块状态中。比如在团队规范里强制：

1. 不允许 `arr[index] = ...`，统一用 `splice`。
2. 不允许直接写 `length`，清空用 `splice(0)`。

你可以封装出一套稳定的数组操作函数，甚至在 code review 中当作硬规则审查。这不是“洁癖”，是为了把不可见的响应式边界变成可见的 API 选择。

## 深层递归与性能：Vue 2 的“深观察”不是免费的

Vue 2 的 `observe` 默认是深度递归，把对象嵌套结构全部走一遍 defineReactive。这意味着：

1. 初始化成本与对象规模成正比，尤其是大列表、深树形结构会带来明显卡顿。
2. 内存开销不可忽视：每个 reactive key 都有 dep、闭包、getter/setter。
3. 某些场景你其实不需要深度响应式（例如只展示一次、或后续只整体替换）。

工程策略上，你需要回答一个更根本的问题：哪些数据必须是响应式的，哪些不必？Vue 2 的常见实践是：

1. 对于大体量数据，用“整体替换”而不是“深度变更”；拿到数据后不要逐字段 patch，而是替换引用触发更新。
2. 对于只读数据，用 `Object.freeze` 表达“不需要响应式”，让 Vue 跳过递归（至少减少后续变更风险）。
3. 对于频繁更新的数值集合，考虑把它们移出响应式系统，用 canvas/虚拟列表/手动渲染等方式承载高频更新。

你可以这样明确表达“我不需要深度响应式”：

```js
data() {
  return {
    // 大型静态配置：冻结后可减少误写与观察成本
    schema: Object.freeze({
      fields: [/* ... */],
      rules: { /* ... */ }
    }),
    // 大列表：整体替换，少做深层 patch
    rows: []
  }
},
methods: {
  async load() {
    const rows = await fetchRows()
    this.rows = rows // 替换引用触发更新
  }
}
```

这类做法的核心不是“优化性能”，而是“把响应式当作一种有限资源来使用”。当你不再把所有东西都交给响应式系统，系统反而更稳定。

## 计算属性与依赖污染：聪明也会带来隐性耦合

计算属性（computed）是 Vue 2 响应式体验中最“优雅”的部分，但它也是大型项目里隐性耦合的温床。原因在于：computed 的依赖来自读取路径，阅读代码时你看不到依赖声明；当 computed 内部读取了某些字段，这些字段变化就会触发 computed 失效并可能连锁触发渲染。

常见的坏味道是：computed 里面做了过多的“业务聚合”，把多个模块状态揉成一个巨大的派生对象，导致任何小变更都引发大范围失效；或者 computed 内部读到了本不该读的响应式字段（例如某个仅用于日志的状态），造成依赖污染。

工程上我建议：

1. computed 只做“结构稳定”的派生，不做“条件分支极多”的业务决策。
2. 把复杂决策拆成多个小 computed，让依赖更局部。
3. 避免 computed 返回全新对象导致的引用抖动（尤其是作为子组件 props），必要时做结构共享或将返回值拆成原子字段。

Vue 2 时代没有自动的依赖可视化工具时，这些“依赖污染”往往是性能问题的根源之一。

## 常见误区：把响应式当成“数据层真理”

### 误区 1：以为“数据变了 UI 就会变”

Vue 2 的响应式只对它能拦截到的写入负责。你用 `obj.newKey = 1`、`arr[i] = x`，写入确实发生了，但 UI 不一定会更新。这个问题不是“框架 bug”，而是你绕开了契约。

### 误区 2：把任意对象塞进 data，然后在各处随意改

当一个对象同时被 Vue 响应式系统和外部逻辑共同持有，并且外部逻辑会直接改它的属性，你很容易出现“写入路径不统一”的问题：有时通过 setter，有时绕开 setter。最终的表现就是：同样的业务动作，有时更新、有时不更新。

解决方案不是“多写 Vue.set”，而是把写入路径收敛：定义状态的唯一写入口（store/action 或模块函数），外部逻辑只能通过入口请求变更。

### 误区 3：把性能问题归咎于“响应式慢”

多数性能问题来自“依赖范围太大”和“变更频率太高”，而不是响应式本身。Vue 2 的响应式是足够快的，但它不是为 60fps 高频流数据设计的。你需要为高频场景选择不同的承载方式。

## 反模式：用 Vue.set“救火”而不治理数据结构

Vue.set 是工具，不是架构。项目里如果到处都是 `Vue.set`/`this.$set`，往往意味着你的数据结构在不断漂移：字段不稳定、接口不稳定、状态 shape 缺乏统一约束。短期看你在“修 bug”，长期看你在把系统变得不可推断。

我更建议把 `Vue.set` 当成边界动作：只允许在少数“字典型数据结构”上使用（比如以 id 为 key 的 map），并在评审中要求作者解释“为什么这个结构必须是字典而不是定型对象”。

## 我的判断：Vue 2 响应式的正确用法是“让边界变成规则”

Vue 2 响应式的价值在于它默认提供了极低的开发成本：你写模板，它帮你追踪依赖；你改数据，它帮你更新 UI。要把这套价值维持到大型项目阶段，你必须付出另一种成本：用规则把边界变显式。

我的实践判断是：

1. 数据结构要尽量定型（固定 shape），把新增 key 变成例外而不是常态。
2. 数组更新要收敛到约定接口（splice 等），把“可观察”变成规范。
3. 对大体量/高频数据要敢于放弃深度响应式，用“整体替换/冻结/非响应式渲染”换确定性。
4. computed 要克制，避免把它当成业务聚合器；依赖范围越小，系统越稳。

做到这些，你会发现 Vue 2 的响应式并没有“失手”，它只是要求你像对待任何一个有边界的系统一样：尊重契约、控制出口、减少偶然性。
