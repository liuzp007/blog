## 先定目标：大型项目里“稳定”指的是可预测、可回滚、可定位

“稳定”常被误解成“少改动”。在大型项目里，少改动不是稳定，甚至是另一种失控：需求还在变、依赖还在变、安全与合规还在变，你不改只是在把风险推迟到不可控的某一天。

我更愿意把稳定定义为三个可操作指标：

1. **可预测**：同样的输入、同样的环境，系统行为可复现；变更的影响面可推断。
2. **可回滚**：上线出现问题，能在分钟级恢复；恢复路径不依赖“神秘专家手工操作”。
3. **可定位**：线上问题能快速归因到模块/提交/配置；定位成本随系统规模增长得尽量慢。

Vue 2 本身不是稳定性的来源；稳定性的来源是围绕 Vue 2 建起来的工程体系。把这句话咬住，你就不会陷入“升不升级”的口水仗，而会把注意力放在真正能让系统稳的抓手上。

## 版本冻结的幻觉：不升级不等于风险归零

很多团队选择长时间维护 Vue 2，会在心理上把它等同于“稳定”：框架不变、行为不变、风险更小。实际情况恰恰相反：当你冻结核心依赖时，风险并不会消失，它会转移并累积在其他地方：

1. 浏览器环境变化：新特性、兼容性策略、性能特征会变化。
2. 依赖生态变化：构建工具、周边库、安全漏洞修复都要求升级。
3. 业务复杂度增长：状态数量、页面数量、协作人数增加，系统更需要治理。
4. 安全合规要求：依赖漏洞、许可证、隐私要求会不断变化。

因此，“维护 Vue 2”的真正工作不是守着一个版本不动，而是构建一套**在不频繁大迁移的前提下持续吸收变化**的能力。

## 稳定性的第一来源：把依赖关系做成“可审计的图”

大型 Vue 2 项目最常见的失控点不是模板，而是依赖关系：组件依赖组件、模块依赖模块、页面依赖工具库、工具库依赖全局状态。依赖不可见就不可治理。

工程上我建议把依赖治理分成三层：

1. **模块边界**：用目录结构与导入规则约束“哪些模块能依赖哪些模块”。
2. **运行时边界**：禁止跨层写入（例如业务组件直接写全局单例、随意读写 localStorage）。
3. **构建边界**：约束打包输出与 chunk 结构，避免引入不可控的体积与加载链路。

一个可落地的做法是：为 `src/` 下的模块制定导入白名单，配合 ESLint 规则或简单脚本校验。即便你不做复杂工具，也可以先从“禁止跨层相对路径”和“禁止从页面直接导入底层实现文件”开始，逐步收敛。

## 稳定性的第二来源：状态治理比组件治理更关键

Vue 2 的组件体系本身并不难维护，难维护的是状态在系统中流动的方式。大型项目里状态失控通常表现为：

1. 状态散落：同一个概念在多个地方各有一份，互相“同步”。
2. 写入口不一致：谁都能改，改完靠 watcher 修补一致性。
3. 语义漂移：一个字段最初代表 A，后来被复用代表 A+B，最终没有人敢删。

在 Vue 2 项目里，最有效的治理手段往往不是“重构组件”，而是建立状态层协议：

1. 明确哪些状态属于组件内部、页面级、领域级共享。
2. 领域级共享状态必须通过 action/mutation 的意图入口修改，禁止随意写。
3. 对关键状态的变更建立日志与回放能力（至少在开发/预发环境）。

示例：把“写入口”集中到 action，组件只表达意图。

```js
// 组件里
methods: {
  onSubmit() {
    this.$store.dispatch('order/submitOrder', { cartId: this.cartId })
  }
}

// store/modules/order.js
export const actions = {
  async submitOrder({ commit }, { cartId }) {
    commit('SUBMIT_START', { cartId })
    try {
      const res = await api.submit(cartId)
      commit('SUBMIT_SUCCESS', { orderId: res.id })
    } catch (e) {
      commit('SUBMIT_FAIL', { message: String(e) })
      throw e
    }
  }
}
```

这段代码的工程价值在于：你能在一个地方观察到状态如何变化、失败如何处理、回滚如何实现。稳定性来自“变更可追踪”，而不是来自“不要改”。

## 稳定性的第三来源：发布与回滚必须成为“默认动作”

大型项目里最怕的是：线上出问题时只能“紧急修复并祈祷”。真正稳定的系统需要把回滚做成默认能力。

这不是 Vue 2 特有问题，但 Vue 2 项目常见的挑战是：历史包袱导致构建输出不稳定、环境变量多、灰度策略缺失，回滚变得昂贵。

一套可落地的发布/回滚策略可以从这些动作开始：

1. 产物可复用：同一 commit 在不同环境产物尽量一致（环境差异通过配置注入而非代码分支）。
2. 灰度与开关：关键功能用 feature flag 包装，允许逐步放量与快速关闭。
3. 版本可追踪：页面能显示当前版本号/commit hash，日志里能关联版本。
4. 回滚可自动：回滚不依赖“重新打包”，而是切换到上一份已存在的产物。

你不必一次做全，但必须开始做，否则“稳定”只能是口号。

## 性能治理：Vue 2 项目稳定性的隐藏支柱

性能问题在大型项目里会变成稳定性问题：卡顿导致误操作、超时导致重复提交、页面白屏导致用户流失。Vue 2 的性能优化如果只靠“加 computed、加 keep-alive”，很快会撞墙，因为性能问题往往来自系统性原因：

1. 依赖范围过大：某个状态变更触发了大量组件更新。
2. 列表渲染过重：大列表/复杂行组件导致渲染成本飙升。
3. 资源加载失控：首屏 chunk 过大，异步 chunk 过碎导致瀑布。

治理性能的关键是“预算化”：把性能从“出了问题再救火”变成“持续约束”。可以落地的预算包括：

1. 首页 JS 体积上限、首屏渲染时间目标。
2. 关键交互的响应时间上限（例如点击到弹窗出现）。
3. 长列表渲染策略（虚拟列表/分页/折叠）。

在 Vue 2 中，最经常被忽略但最有效的动作之一是：把高频更新的部分从响应式系统里移出去（例如用 canvas 或 requestAnimationFrame 手动更新），让 Vue 只负责低频结构更新。稳定性来自“系统承载能力匹配”，不是来自“优化某个函数”。

## 可观测性：没有可观测，就没有可维护

大型项目维护 Vue 2，经常会遇到一种“最难的 bug”：用户说卡了、白了、错了，但你在本地复现不了。此时靠工程纪律而不是靠英雄主义。

最低限度的可观测性应该包括：

1. 错误边界与全局异常捕获：能收集未处理异常、资源加载失败、Promise rejection。
2. 关键链路埋点（不引入第三方分析系统也可以做日志上报，但需符合你的项目边界）：至少记录页面加载成功/失败、关键 API 成功/失败。
3. 性能指标采样：首屏、路由切换、长任务等。

Vue 2 自己的错误边界能力有限（不像 React 那样），但你仍然可以通过全局 errorHandler、window.onerror、unhandledrejection 等做基本兜底，并把信息关联到版本号与路由。

## 故障演练：稳定性靠“演过一次”而不是靠“相信不会发生”

很多团队把稳定性建设理解为“加监控、加告警”。这些很重要，但如果你没有做过故障演练，真正出问题时你会发现：你知道发生了什么，但你不知道该怎么恢复。

我建议把“演练”当成一个产品能力去建设，至少覆盖三类场景：

1. 资源加载失败：CDN 局部不可用、某个 chunk 404、缓存命中旧版本导致运行时报错。
2. API 异常：关键接口超时、返回结构变化、幂等性失效导致重复提交。
3. 版本回滚：新版本引入严重 bug，需要在分钟级切回上一版本。

每一类演练都应该产出一个“最小恢复路径”：谁负责判断、切哪个开关、怎么验证恢复成功、如何防止二次扩散。演练的价值不是证明你强大，而是把“恢复路径”从人的脑子里搬到流程和脚本里。

下面给一个非常务实的“产物版本标识 + 快速验证”做法：在构建时注入版本号，页面底部或 debug 面板能看到；同时在日志上报中带上版本号和路由，定位才有锚点。

```js
// build-time: 注入版本号（示意）
window.__APP_VERSION__ = '2026-03-24-<commit>'

// runtime: 错误上报携带版本与路由
window.addEventListener('unhandledrejection', (e) => {
  reportError({
    type: 'unhandledrejection',
    message: String(e.reason || ''),
    version: window.__APP_VERSION__,
    route: location.pathname + location.search + location.hash
  })
})
```

注意：这里的重点不是“上报到哪里”，而是你要在系统里形成一条可追踪链路：用户看到的问题 -> 日志能关联到版本 -> 版本能映射到 commit -> 回滚能切回旧产物。

## 依赖收敛的落地细则：把“可以依赖”变成工具能检查的规则

很多人认同“要有边界”，但边界经常停留在 PPT。大型项目要把边界落地，需要两个条件：

1. 规则足够简单，团队能执行。
2. 规则可被工具检查，否则会被忙碌吞没。

一个从零开始也能落地的方案是：把 `src/` 目录按层级拆分（例如 `pages/`、`features/`、`components/`、`utils/`、`store/`），然后定义“禁止逆向依赖”的硬规则。比如：

1. `utils` 不能依赖 `components/features/pages`。
2. `components` 不能依赖 `pages`。
3. `features` 可以依赖 `components/utils/store`，但不能依赖 `pages`。

你不需要一开始就做复杂的架构图，先把最危险的逆向依赖堵住。然后用 ESLint 或简单脚本做检查，把违规变成 CI 失败，而不是评审“建议”。

即便只是伪码式的检查脚本，也能立刻提升稳定性，因为它让依赖扩散变难：

```js
// 伪码：扫描 import 路径，发现逆向依赖就报错
if (from.startsWith('src/utils/') && to.startsWith('src/pages/')) fail()
if (from.startsWith('src/components/') && to.startsWith('src/pages/')) fail()
```

稳定性建设经常被低估的一点是：把规则工具化之后，团队的注意力才能从“反复提醒”转移到“解决真正问题”上。

## 常见误区：把维护工作等同于“修 bug”

### 误区 1：稳定性 = 修 bug 的速度

修得快不等于稳。真正稳的系统会让 bug 变少、变浅、变容易定位。你需要把一部分精力投入到“让 bug 不再出现”的机制上，例如写入口收敛、回滚能力、依赖边界。

### 误区 2：只优化组件，不治理状态与依赖

你可以把某个组件重构得很漂亮，但如果它依赖的状态仍然全局可写、依赖链仍然不可见，系统仍然会在别处崩。维护的优先级应当从“结构性风险”开始，而不是从“局部优雅”开始。

### 误区 3：把“升级”当作唯一解决方案

升级到 Vue 3 能带来很多收益，但如果你没有治理能力，升级只会把问题搬家：耦合仍然耦合、发布仍然不可回滚、性能仍然不可预算。升级是手段，不是稳定性的来源。

## 反模式：用“全局技巧”掩盖结构问题

大型 Vue 2 项目里常见的反模式是：当系统出现不一致或性能问题时，靠全局技巧修补，而不改变结构。

1. 到处加 watcher 维持状态一致性，最后形成更新链地狱。
2. 到处加 nextTick 试图修复时序，最后把渲染调度变成偶然。
3. 到处用 event bus 解决跨层通信，最后依赖关系不可审计。
4. 把所有东西塞进 Vuex，最后全局状态成为耦合垃圾场。

这些技巧在局部问题上可能有效，但在系统层面会持续放大维护成本。你需要用“边界治理”替代“全局技巧”。

## 我的判断：维护 Vue 2 的关键是建立“演进能力”，而不是维持“静止”

我支持长期维护 Vue 2，但前提是你把目标从“尽量不动”改成“可持续演进”。我的判断标准是：

1. 依赖关系可审计，模块边界能被工具或规范约束。
2. 状态写入口收敛，关键状态变更可追踪。
3. 发布与回滚是默认动作，线上事故不靠英雄主义。
4. 性能有预算，资源加载与更新范围可控。
5. 具备基本可观测性，定位成本可接受。

当这些能力成立时，你维护的是一个“可治理的系统”，Vue 2 只是其中一个实现细节；当这些能力不成立时，即便你升级到更先进的框架，系统仍然不会稳定。稳定性的来源从来不是“不升级”，而是工程体系。
