很多团队讨论 Vue 2 升级 Vue 3，第一反应是“语法改多少”“改动量有多大”“能不能自动化”。这些问题当然重要，但它们常常把团队带到错误的战场：你们会在 `setup()`、`ref()`、`reactive()`、`defineComponent()` 的细节里耗掉 80% 的时间，最后发现线上回归、协作效率下降、组件抽象更混乱，迁移变成“版本号换了，复杂度没变”。

我想把观点说得更尖一点：**Vue 2 到 Vue 3 的迁移，本质不是语法迁移，而是心智迁移。**它要求团队把“我在组件里写逻辑”的习惯，迁移成“我在系统里建边界”的习惯；把“框架替我处理很多隐式事情”的舒适区，迁移成“依赖显式、生命周期显式、抽象可组合”的工程表达。

如果你把迁移目标定义为“能跑起来”，你会获得一个能跑但难以治理的 Vue 3 项目；如果你把迁移目标定义为“把系统复杂度降回可控”，你才会知道该保留什么、替换什么、推迟什么，以及什么时候该停下来补齐能力，而不是继续搬运代码。

本文给出一条更工程化的迁移路径：它不追求一次性最优，而追求**风险可控、增量可验证、成本可解释**。同时我会明确指出一些常见反模式，帮助你避免把迁移做成“组织性事故”。

## 先定义“迁移完成”的标准：不是版本号，而是成本曲线

“迁移完成”的定义决定了你的路线。只看版本号，意味着你会为了尽快升上 Vue 3 而做大量折衷；只看功能上线，意味着你会把迁移变成永久拖延的背景噪音。更合理的方式是定义一组可以被验证的目标，覆盖四类成本：

1. 交付成本：迭代速度、回归周期、线上发布失败率。
2. 维护成本：代码阅读成本、组件复用成本、依赖升级成本。
3. 质量成本：缺陷密度、线上问题定位时长、可观测性（日志、错误边界、回滚路径）。
4. 组织成本：新同学上手时长、团队对“正确写法”的一致性、评审争议密度。

你不需要把这些都量化得很精确，但至少要给出可对齐的口径。例如：

- “迁移完成”不是“页面都换成 Vue 3”，而是“核心业务链路在 Vue 3 下运行且 e2e 用例稳定通过，且新功能只在 Vue 3 侧开发”。
- “迁移完成”不是“Options API 都改成 Composition API”，而是“复杂模块的抽象方式统一，不再依赖隐式 mixin/全局副作用，逻辑复用以 composable 为主”。

当你把迁移当作成本曲线优化，你会自然做出两个关键决策：

1. 哪些东西可以兼容层过渡，哪些必须重做。
2. 哪些回归风险必须前置验证，哪些可以后置清理。

## 心智迁移 1：从 defineProperty 到 Proxy，不是能力变强，而是语义更清晰

Vue 2 的响应式让人“写起来爽”，因为它把很多约束藏在框架里：对象的属性通过 getter/setter 收集依赖，数组方法被 patch，渲染更新通过 watcher 队列批处理。代价是：你必须记住一批“坑位”，例如新增属性需要 `Vue.set`、数组下标赋值不触发、某些深层对象的变更追踪不直观等。

Vue 3 通过 Proxy 把“能不能追踪”从“靠补丁”变成“语义上的统一”：访问就 track，变更就 trigger。很多人看到这里就得出结论：“Vue 3 更强了，所以迁移就是把 API 替换掉”。这仍然是把注意力放错了地方。

更重要的变化是：**你开始被迫把“数据形态”和“依赖边界”当作工程设计对象。**在 Vue 2 时代，你可能习惯在任意地方把一坨对象塞进 data，然后在各个组件里随意读写；Vue 3 时代，这种做法会让依赖关系变得更隐蔽、更难治理，因为一切都能被追踪，一切也都可能触发更新。

迁移时你要问的不是“这个对象 reactive 之后会不会更新”，而是：

1. 这个状态应该被谁拥有？（组件、composable、store、服务层）
2. 哪些字段属于“领域状态”，哪些只是“UI 状态”？（避免把 UI 细节扩散到共享状态）
3. 依赖集合是否稳定？（避免 render 依赖随条件分支频繁变动造成抖动）

一个典型迁移点是“把 Vue 2 的 data 巨对象拆开”。在 Vue 3 里，你更应该把状态按职责拆成多个更窄的响应式来源，而不是让一个巨大的 reactive 对象承担所有语义。

```ts
// Vue 3 里更可治理的做法：按职责拆分状态来源
import { reactive, ref, computed } from 'vue'

const query = reactive({ keyword: '', page: 1, pageSize: 20 })
const ui = reactive({ loading: false, error: '' })
const items = ref<Array<{ id: string; title: string }>>([])

const hasMore = computed(() => items.value.length < query.page * query.pageSize)
```

这不是“写法升级”，是边界升级：你在把“会一起变化的东西”放一起，把“不同生命周期的东西”拆开，降低更新耦合。

## 心智迁移 2：从 mixin 到 composable，变化的是抽象单位与依赖表达

Vue 2 大型项目里，mixin 曾经是最常见的逻辑复用手段之一。它的问题不是“功能不够”，而是它把依赖变成了隐式的：你在组件里看不到 mixin 注入了什么数据、方法、生命周期；更糟糕的是多个 mixin 叠加时会产生命名冲突与时序冲突，最终只有少数人敢动。

Vue 3 的 Composition API 让逻辑复用回到更接近“普通函数”的抽象形态：composable 是显式调用，依赖可以通过参数显式传入，返回值是明确的能力集合。迁移的难点不在于“把 mixin 改成函数”，而在于你必须重新定义抽象的边界：

1. composable 要解决哪一类问题：领域能力、UI 交互、数据访问还是副作用编排？
2. composable 的输入输出是什么：输入越多越像工具函数，输出越多越像小型框架。
3. composable 的生命周期如何管理：watch/watchEffect 的 cleanup 由谁负责，副作用是否可被测试？

如果你只是把 mixin 的内容原封不动搬进一个 `useXxx()` 函数，再在组件里调用，你大概率会得到更难读的代码：因为你同时失去了 Options API 的结构化约束（data/methods/computed），也没有建立新的抽象约束（输入输出、依赖方向）。

迁移的正确姿势是：**先把 mixin 里隐藏的依赖显式化，再决定它应该变成 composable 还是 store 模块，还是干脆应该回到组件内。**

```ts
// 反例：把 mixin 直接搬进 composable，依赖仍然隐式
export function useBadThing() {
  // 这里偷偷读了全局 store、路由、以及 window 状态
  // 组件看不到依赖，测试也难做
}

// 更可控：把依赖显式参数化
export function useSearch(params: {
  fetcher: (q: string) => Promise<string[]>
  debounceMs?: number
}) {
  // ...
}
```

这里的核心不是“函数更优雅”，而是“依赖可见”。当依赖可见，评审和排障的成本会显著下降。

## 心智迁移 3：从“生命周期钩子堆叠”到“副作用编排”，你要学会管理 cleanup

Vue 2 的生命周期钩子让人容易把逻辑写散：mounted 里绑事件、watch 里触发请求、beforeDestroy 里清理，跨多个 mixin 后更是难以追踪。Vue 3 的 `onMounted`、`onUnmounted`、`watch`、`watchEffect` 并不会自动帮你变好，它只是提供了更好的编排能力。迁移时最容易出问题的是 cleanup：

1. 事件监听、定时器、订阅如果不清理，会造成泄漏或重复执行。
2. watch 的依赖如果过宽，会造成不必要的副作用触发，甚至引入竞态条件。
3. watchEffect 的“自动依赖收集”如果没有边界，会把不该成为依赖的东西也绑进来。

在 Vue 3 里我更推荐把副作用写成“可取消的任务”，并把取消逻辑与触发逻辑放在同一个作用域内。这样在迁移时更容易验证行为一致。

```ts
import { ref, watch, onUnmounted } from 'vue'

export function useCancelableRequest<T>(run: (signal: AbortSignal) => Promise<T>) {
  const data = ref<T | null>(null)
  const error = ref('')
  const loading = ref(false)

  let ctrl: AbortController | null = null
  const exec = async () => {
    ctrl?.abort()
    ctrl = new AbortController()
    loading.value = true
    error.value = ''
    try {
      data.value = await run(ctrl.signal)
    } catch (e: any) {
      if (e?.name !== 'AbortError') error.value = String(e?.message || e)
    } finally {
      loading.value = false
    }
  }

  onUnmounted(() => ctrl?.abort())
  return { data, error, loading, exec }
}
```

这类抽象在 Vue 2 里也能做，但 Vue 3 的组合式写法更自然。迁移的重点是：把“副作用散落在各处”收敛成“有边界、可取消、可复用”的能力。

## 生态迁移：你以为在迁框架，其实在迁“依赖图”

Vue 2 项目迁移的真实阻力，往往来自生态：

1. 路由、状态管理、UI 组件库是否支持 Vue 3。
2. 自研插件（prototype 注入、全局指令、全局 filter）是否能平滑迁移。
3. 构建工具链（webpack/vite）与代码规范是否需要同步调整。

很多团队犯的错误是把生态迁移当作“顺手改掉”：框架升级过程中顺便换路由、顺便换 store、顺便换构建工具、顺便换 UI 库。最后任何一个回归都无法定位原因，因为你同时更换了太多变量。

更可控的方式是明确迁移顺序：先稳定住“应用运行的基本壳”，再逐步替换内部能力。举例：

1. 先把全局注入从 `Vue.prototype` 迁到 `app.config.globalProperties`（并建立类型约束）。
2. 再把全局 filter 替换为显式函数或 computed（避免模板隐式依赖）。
3. 再处理 UI 组件库与指令的迁移（这类回归可视化强，适合做 e2e）。

```ts
// Vue 2:
// Vue.prototype.$http = httpClient

// Vue 3:
import { createApp } from 'vue'
import App from './App.vue'
import httpClient from './httpClient'

const app = createApp(App)
app.config.globalProperties.$http = httpClient
app.mount('#app')
```

这段代码的意义不在“写法换了”，而在“全局依赖是否可追踪”：你可以逐步收敛 `$http` 的使用范围，甚至未来替换为显式注入（provide/inject）或服务层。

## 回归策略：把迁移当作“行为合同”验证，而不是“UI 对齐”

迁移最可怕的不是显性的报错，而是隐性的行为偏移：某个 watch 触发时机变了、某个 computed 的依赖集合变了、某个组件更新被更频繁地触发，最终表现为“偶现 bug”“线上卡顿”“部分用户复现不了”。

所以迁移时最重要的工程动作之一，是建立“行为合同”（Behavior Contract）：

1. 关键链路的输入输出合同：在同样输入下，输出结构、错误码、加载状态是否一致。
2. 关键交互的时序合同：点击 -> loading -> 成功/失败 -> UI 状态是否遵循既定时序。
3. 关键缓存的失效合同：路由切换、筛选条件变化时，缓存是否正确失效。

这类合同最好用自动化覆盖，而不是靠人工点点点。你不需要一下子把覆盖率做得很高，但要让它足够“抓住大事故”。实践上我更建议：

1. 对核心链路做 e2e（最少但最硬）。
2. 对复杂逻辑抽象做单测或契约测试（验证边界条件）。
3. 对组件层少做 snapshot，多做行为断言（避免噪音）。

迁移期间评审的关注点也要变化：不要只看“语法是否正确”，要看“行为合同是否更清晰、更容易被验证”。

## 常见误区与反模式：把迁移做成组织性回归

这一段我会写得直接一些，因为很多团队的失败不是技术原因，而是反模式叠加。

### 反模式 1：Big Bang，一次性全量替换

一次性全量替换的诱惑来自“短痛”：看似只要熬过一段时间就结束。但实际工程里，Big Bang 会制造三类灾难：

1. 变量太多，回归无法定位。
2. 队伍被分裂成“迁移派”和“业务派”，沟通成本爆炸。
3. 迁移周期一旦拉长，组织耐心耗尽，最后只能降目标“能跑就行”。

除非你的应用极小、依赖极少、上线风险极低，否则 Big Bang 几乎一定是错误策略。

### 反模式 2：把 Options API 机械翻译成 Composition API

“机械翻译”让你感觉进度很快：data -> ref/reactive，computed -> computed，methods -> function。但它会留下更大的技术债：你失去了 Options API 的组织结构，又没有建立新的抽象边界，最终组件会变成一个更难维护的脚本文件。

正确做法是：把复杂逻辑抽成 composable，把 UI 细节留在组件，把共享状态交给 store 或服务层。不要把所有东西都塞进 `setup()`。

### 反模式 3：继续依赖全局副作用（event bus、隐式注入、模板 filter）

Vue 3 并不会自动消灭这些副作用，甚至会让它们更难追踪，因为逻辑更分散。迁移中如果不主动收敛全局副作用，你只是把旧问题搬到了新框架里。

### 反模式 4：把迁移当作“升级工具链”的借口

框架升级期间换构建工具、换 lint、换格式化、换目录结构，容易把团队拖进无意义的消耗。工具链升级当然有价值，但它必须服务于迁移目标，且要分阶段做，不要把风险叠加在同一个发布窗口。

## 一条可执行的分阶段路线：先搭桥，再分流，最后拆桥

我推荐的迁移路线可以概括为三句话：先搭桥、再分流、最后拆桥。

### 阶段 0：盘点与冻结“隐式接口”

在动代码之前，先盘点以下隐式接口：

1. 全局注入：prototype 上挂了什么（http、logger、auth、feature flags）。
2. 全局约定：filter、mixin、directive、全局组件。
3. 跨模块耦合：event bus、直接访问 store 模块内部结构、直接依赖路由实现细节。

这一步的产物不是文档，而是“迁移清单 + 风险列表”。你要知道哪些东西一动就会炸，哪些可以最后再动。

### 阶段 1：先把“平台能力”迁移到可替换的壳

平台能力包括：路由壳、状态壳、网络壳、权限壳、错误边界、埋点（如果你们有）等。把这些东西抽到明确的入口处，让组件不再靠隐式全局访问。你不需要一次性消灭所有全局访问，但要让新代码不再新增隐式依赖。

### 阶段 2：按业务域分批迁移，建立可重复的迁移模板

不要按“页面”迁移，而要按“业务域”迁移：一个域内的组件共享状态、共享交互、共享依赖，迁移时可以一起验证。每迁移一个域，就总结模板：

1. 组件层如何组织（哪些用 Options，哪些用 Composition）。
2. 共享逻辑如何抽（composable 还是 store）。
3. 依赖如何注入（参数、provide/inject、globalProperties）。
4. 回归如何验证（e2e 覆盖哪些链路）。

当模板稳定后，迁移速度会明显提升，且回归风险可控。

### 阶段 3：拆桥与清债

最后一阶段才是“清理”：移除兼容层、替换遗留全局副作用、统一抽象方式、补齐类型与测试。很多团队失败在这里：他们在迁移中期就开始大规模清理，导致迁移速度下降，组织耐心耗尽。正确做法是先把业务迁过去，让新能力可用，然后再集中清债。

## 我的判断：迁移成功与否，取决于你是否愿意把“隐式协作”变成“显式契约”

我对 Vue 2 -> Vue 3 迁移的最终判断很明确：

1. **不要把迁移当作语法升级**。真正要升级的是边界、依赖表达与回归机制。
2. **不要追求全量改造的纯粹性**。增量迁移不是妥协，而是风险控制手段；只要你的“新代码不再制造旧债”，增量是可收敛的。
3. **迁移的核心胜利不是“组件都写 setup 了”**，而是“系统的隐式接口减少了，行为合同更清晰了，排障与迭代成本下降了”。

如果你把迁移当作一次“把隐式协作改成显式契约”的工程重构，你会在 Vue 3 里获得长期收益；如果你只把它当作“换一套写法”，你会得到一个更现代但同样混乱的代码库，而且它更难救，因为你会以为自己已经“升级过了”。
