## 先把框架放下：通信的本质是“依赖方向”和“变更所有权”

很多团队讨论 Vue 2 组件通信，会陷入“这次用 event 还是用 bus”这种局部选择题。真正决定系统能不能长大的是两件事：

1. **依赖方向**：谁依赖谁，谁能看见谁，谁有权发起变更。
2. **变更所有权**：数据由谁拥有，谁负责定义它的语义，谁负责保证一致性与回滚。

props/event/bus/store 只是把这两件事编码成不同的约束：

1. props 强约束依赖方向（父到子）和数据所有权（父拥有，子消费）。
2. event 强约束变更发起方式（子向父“请求”或“通知”）但不强约束数据结构。
3. bus 弱约束一切：任何人都能发消息，任何人都能订阅，依赖方向变得隐形。
4. store 强约束“共享可见性”，但对“谁拥有语义”需要你用模块与约定补齐，否则就会把耦合集中成一个更难治理的点。

所以这篇文章的主观点很简单：**通信不是越灵活越好，通信越灵活，治理成本越高；而大型项目的关键是把“灵活”变成“有边界的灵活”。**

## props：最便宜的通信方式，但它要求你尊重“不可变契约”

### props 的优势不是“传参方便”，而是“把所有权写在依赖图上”

props 是 Vue 2 里最便宜、最可靠的通信方式。它便宜的原因并不是语法短，而是它天然满足了两个重要的工程约束：

1. **依赖方向固定**：父组件依赖子组件的接口（props 定义），子组件不依赖父组件的内部状态。
2. **所有权清晰**：父组件拥有数据，子组件只消费，消费端不负责一致性。

这两条会直接降低协作成本：你看子组件的 props 声明，就能知道它需要哪些输入；你改父组件的数据结构，就知道影响范围是什么。

### props 的真实边界：当你开始“透传一切”，其实是在逃避建模

props 最大的问题也在它的优势：太容易用，于是团队会不自觉把它当作“万能注入器”。

典型表现是：

1. 父组件把一整个 `form`、`pageState`、`config` 对象透传给深层子组件。
2. 子组件为了方便，从大对象里随便取字段。
3. 过几个月，没人知道子组件到底依赖哪些字段；父组件每次改对象 shape 都像拆炸弹。

如果你的子组件依赖是“结构性的”，你应该把它拆成更小、更语义化的 props；如果依赖是“操作性的”，你应该用 event 把“我想做什么”上报，而不是传函数把父组件逻辑塞进子组件。

下面是一段推荐风格：把 props 变成语义明确、稳定的接口。

```js
// ChildUserCard.vue (Vue 2 Options API)
export default {
  name: 'ChildUserCard',
  props: {
    userId: { type: String, required: true },
    displayName: { type: String, required: true },
    avatarUrl: { type: String, default: '' },
    isSelected: { type: Boolean, default: false }
  }
}
```

对比“透传整个 user 对象”，这段代码的价值在于：它把组件对外依赖变成了稳定契约。即便未来 user 的内部结构变了，这个组件也未必需要改。

### 子组件改 props 是反模式：它会同时破坏所有权与时序

Vue 2 会在开发模式下警告你“避免直接修改 prop”。很多人把它当作 lint 级别的建议，但它其实是架构红线。

子组件直接改 props 的后果不是“会不会生效”这么简单，而是它会让系统失去可推断性：

1. **所有权被偷走**：父组件以为自己拥有数据，实际上子组件在暗中改。
2. **回放不可能**：你无法从父组件的输入推导出子组件的状态，因为子组件有隐式写入。
3. **时序难以稳定**：多个子组件可能同时写同一份数据，最后写入顺序变成偶然。

如果你确实需要子组件内部编辑 props 的值，正确的工程手法是：在子组件内部创建一份本地副本，并通过 event 把变更提交回父组件（或 store）。

## event：最合适的“上行”通道，但要把 event 当作“领域事件”而不是“回调函数”

### event 的优势：把“动作”与“状态”分离

event 的本质是：子组件告诉父组件“发生了什么”或“我想做什么”。它不应该把父组件的逻辑倒灌进子组件，而应该表达子组件视角下的动作语义。

一个健康的 event 设计通常具备：

1. 事件名表达领域语义，而不是 UI 实现细节。
2. payload 是最小必要信息，而不是父组件内部对象引用。
3. 子组件不关心父组件如何处理结果，父组件可以同步处理，也可以异步处理。

示例：子组件只发出“选择变更”，父组件决定怎么落到状态。

```html
<!-- ChildTagPicker.vue -->
<template>
  <div>
    <button
      v-for="t in tags"
      :key="t"
      :class="{ active: value.includes(t) }"
      @click="toggle(t)"
    >
      {{ t }}
    </button>
  </div>
</template>

<script>
export default {
  name: 'ChildTagPicker',
  props: {
    tags: { type: Array, required: true },
    value: { type: Array, default: () => [] } // 当前选中
  },
  methods: {
    toggle(tag) {
      const next = this.value.includes(tag)
        ? this.value.filter(x => x !== tag)
        : this.value.concat(tag)
      this.$emit('change', next) // 领域语义：选择变化
    }
  }
}
</script>
```

父组件接收 `change`，决定怎么更新 store 或本地状态。子组件不传“更新函数”，也不直接写 store。

### event 的边界：当你开始“事件风暴”，你可能在用 event 模拟全局状态

event 适合上行，但它也有边界：当你需要跨层级、跨路由、跨模块沟通时，event 会演化成“多点连线”，导致：

1. 事件链路很长：子组件 emit，父组件再 emit，一层层向上冒泡。
2. payload 逐层变形：不同层级加字段、删字段，最终语义漂移。
3. 处理位置不稳定：某个事件到底在哪一层处理，变成协作成本。

这时你应该暂停，问一个结构性问题：这是“局部动作”还是“共享状态”？如果是共享状态，就该考虑 store；如果是跨模块协作，就该考虑更明确的“领域服务层”（模块函数）而不是继续冒泡 event。

## bus：为什么它诱人，以及为什么它会在规模化时“背叛你”

### bus 的诱惑：不用建模就能连通

bus（事件总线）在 Vue 2 时代非常常见：`const bus = new Vue()`，然后全局 `bus.$on/$emit`。它的诱惑是显而易见的：任何组件都能发消息，任何组件都能订阅，仿佛瞬间解决了“跨层级通信”。

典型写法如下：

```js
// bus.js
import Vue from 'vue'
export const bus = new Vue()

// A.vue
import { bus } from './bus'
bus.$emit('toast', { type: 'success', message: 'ok' })

// B.vue
import { bus } from './bus'
export default {
  created() {
    bus.$on('toast', this.onToast)
  },
  beforeDestroy() {
    bus.$off('toast', this.onToast)
  },
  methods: {
    onToast(payload) { /* ... */ }
  }
}
```

这段代码在小项目里确实“跑得起来”。但它的问题不是“会不会内存泄漏”，而是它让依赖关系变得不可见：谁在发 toast？谁在听 toast？有哪些 payload 形状？这些都不在静态结构里，靠搜索字符串和运行时经验。

### bus 的真实代价：隐式依赖 + 时序不确定 + 难以测试

bus 的三个成本会在项目变大时叠加爆炸：

1. **隐式依赖**：组件看起来很“独立”，实际上依赖一堆全局事件。你删一个 `$emit` 可能让远处某个功能静默失效。
2. **时序不确定**：事件处理顺序取决于订阅注册顺序，订阅注册顺序又取决于组件创建顺序。你很难给出可稳定复现的推断。
3. **难以测试**：单测需要构建一个“事件环境”，并且要清理订阅；而集成测试中事件链路不透明，定位成本极高。

你可以把 bus 当作“隐式的全局变量”。工程上只要你承认它是全局变量，你就会对它更克制：全局变量不是不能用，但要非常少，并且要有明确的治理规则。

### bus 的合理边界：它只该用于“基础设施级别”的一次性广播

如果你非要用 bus，我建议把它限制在两类场景：

1. 与 UI 基础设施相关且不承载业务语义的广播：例如全局 toast、全局 loading、全局快捷键。
2. 需要跨路由、跨模块，但有明显“一对多通知”形态且 payload 极小且稳定的事件：例如“语言切换”“主题切换”。

并且要把它做成一个受控模块：事件名集中定义、payload 结构集中定义、订阅统一封装，避免组件里散落 `$on/$emit`。

## store（Vuex）：集中并不等于清晰，清晰来自“模块边界 + 变更协议”

### store 解决的是“共享可见性”，不是“业务建模”

很多团队引入 Vuex 是因为“通信太乱了”。这确实是一个合理动机：store 能把跨组件共享的数据放到一个可观察的中心，让任何组件都能读到一致的状态。

但 store 的常见陷阱是：你把状态集中起来了，却把语义也混在一起了。最终的结果是：原本散落在组件里的耦合，被搬到了 store 里，变成更难拆的耦合。

如果你希望 store 真正降低复杂度，你需要做两件事：

1. **模块化**：按领域拆模块，避免一个巨大 root state。
2. **变更协议**：约束 mutation/action 的语义，避免随便写 state。

下面给出一个“协议化”的示例：action 表达业务意图，mutation 只表达状态变更，payload 结构稳定且可追溯。

```js
// store/modules/cart.js
export const state = () => ({
  items: [] // { skuId, count }
})

export const mutations = {
  ADD_ITEM(state, { skuId, count }) {
    const found = state.items.find(x => x.skuId === skuId)
    if (found) found.count += count
    else state.items.push({ skuId, count })
  },
  SET_COUNT(state, { skuId, count }) {
    const found = state.items.find(x => x.skuId === skuId)
    if (found) found.count = count
  }
}

export const actions = {
  addToCart({ commit }, { skuId, count }) {
    // 业务意图：加入购物车（可在这里做校验/埋点/异步）
    commit('ADD_ITEM', { skuId, count })
  },
  setCartCount({ commit }, { skuId, count }) {
    commit('SET_COUNT', { skuId, count })
  }
}
```

关键点在于：组件发起的是 `addToCart` 这类业务意图，而不是“我想把 items[3].count++”。当你把“意图”变成入口，系统才有机会治理。

### store 的边界：不要把所有状态都放进去

store 不是“更强的 data”，它是“更贵的共享状态”。贵在：

1. 更难拆：一旦多个模块依赖同一份状态，迁移成本上升。
2. 更难理解：全局可见意味着全局可写（如果没有协议），风险上升。
3. 更难测试：你需要构建 store 环境，mock action、副作用、插件等。

因此，大型项目里合理的分层通常是：

1. **局部 UI 状态**（弹窗开关、输入框内容、临时 hover）留在组件内。
2. **页面级状态**（筛选条件、分页、当前 tab）可留在页面组件或路由 query 中。
3. **领域共享状态**（用户身份、权限、购物车、全局配置）进入 store，并且必须协议化。

如果你把临时 UI 状态也塞进 store，你会让每一个微小交互都变成全局事件，既影响性能，也影响可维护性。

## 通信方案的升级路径：从局部到全局，靠“语义稳定”而不是“技术切换”

通信方案不应该一次选型定终身，它更像一条升级路径：当系统规模、团队规模、变更频率发生变化时，你应该允许某些通信方式升级。

我建议一个可落地的升级原则：

1. **默认用 props/event**：只要还能用清晰的父子关系表达，就不要引入全局机制。
2. **跨层级但同一页面：优先组合组件/提升状态**：先考虑把共享状态提升到共同祖先，再通过 props/event 分发。
3. **跨页面/跨模块：引入 store，但先模块化与协议化**：不要先把状态搬进 store，再想模块边界；应该先定义模块边界与意图入口。
4. **bus 只保留基础设施广播**：把业务事件从 bus 迁出，避免隐式依赖扩散。

这条路径的关键是“语义稳定”：你在升级通信方式时，尽量保持组件接口语义不变（props/event 不变），把变更压缩在状态层（store）或基础设施层（toast/service）。

## 常见误区：通信问题本质上是“建模问题”，不是“写法问题”

### 误区 1：把 props 当成依赖注入容器

透传大对象、透传函数、透传 store 实例，本质上都是把父组件内部建模泄漏到子组件。短期看省事，长期看你在制造“隐式接口”，让组件复用与重构成本飙升。

### 误区 2：事件名随手起，payload 随手塞

event 如果没有命名规范和 payload 约束，会很快变成“弱类型 RPC”。你会看到各种 `update`、`change`、`submit`，payload 里塞着父组件内部对象引用。最后 event 变成“隐藏的双向绑定”，把边界打穿。

### 误区 3：用 bus 解决跨层级，直到 bus 变成系统的神经网络

当你发现调试靠“全局搜索 toast 字符串”，并且每次改事件名都害怕引发连锁反应，你就已经在为 bus 的隐式依赖付费了。此时继续修补订阅/注销，只是在延长债务生命周期。

### 误区 4：引入 Vuex 就万事大吉

Vuex 只是把通信从“组件之间”转移到“组件与 store”。如果你不定义模块边界、不定义意图入口、不限制写入方式，最终会得到一个更大的“全局 bus”，只不过换成了 mutation/action 名字。

## 反模式：双向数据流的“伪同步”与跨层写入

下面列几个在 Vue 2 大型项目里非常常见、但危害很大的反模式。它们的共同点是：破坏所有权，制造隐式时序。

1. **子组件直接改父组件对象引用**：父传 `form`，子直接 `form.name = ...`。看似没改 props（改的是对象内部），但所有权已经被偷走。
2. **事件回调里做链式写入**：子 emit A，父处理 A 后又 emit B，再由另一个组件处理 B；链条越长，越难调试。
3. **bus 承载业务事件**：把“订单创建成功”“权限变更”“数据刷新”做成 bus 广播，订阅者散落各处，任何一次改动都无法确定影响面。
4. **store 里塞 UI 临时状态**：比如把“某个弹窗的输入内容”放进 store，让每次输入都变成全局 mutation，产生无意义的耦合与性能开销。

一个“跨层写入”的典型坏例子如下：

```js
// 父组件
<ChildEditor :form="form" />

// 子组件（反模式）
export default {
  props: { form: Object },
  methods: {
    onInputName(v) {
      // 看起来改的是 form 内部字段，不会触发 Vue 的 prop 警告
      // 但你已经把所有权从父组件偷走了
      this.form.name = v
    }
  }
}
```

这类代码短期很“丝滑”，长期会让你无法回答“是谁在什么时候改了这个字段”。修复方式不是加注释，而是重建边界：要么用 `v-model`/`input` event 把变更上报，要么把编辑状态内聚到子组件，最后一次性提交。

## 一套可执行的边界清单：让 code review 有抓手

如果你希望团队能持续做对选择，最好把原则翻译成 code review 规则。这里给一套我认为可以落地的清单：

1. 子组件禁止直接修改 props 或 props 引用对象的内部字段（除非明确声明为“可变对象协议”，且极少见）。
2. event 名必须表达业务语义，禁止使用模糊的 `update`/`change` 覆盖所有动作（可以允许局部 `input/change` 但要限定范围）。
3. event payload 禁止携带父组件内部对象引用（尤其是 store state 的引用），只传最小必要信息。
4. bus 禁止承载业务事件，只允许基础设施广播，并集中定义事件名与 payload 结构。
5. store 的 mutation/action 必须具备模块前缀与意图语义，禁止出现“通用 setter”式 mutation（例如 `SET_STATE` 接受任意 key）。
6. store 禁止存放局部 UI 临时状态，除非它确实是跨页面共享且具有稳定语义的领域状态。

清单的价值在于：它把“设计判断”变成“执行动作”，让团队能在日常评审里持续纠偏。

## 我的判断：Vue 2 的通信体系要以“边界感”为第一原则

Vue 2 时代最容易犯的错，是把通信当成“技术选项”，而不是“架构契约”。当项目规模不大时，bus 和随手透传对象确实能让你更快交付；但规模一上来，你付出的不是“多写几行代码”，而是“系统可推断性”的崩塌：你不知道依赖图是什么，你不知道谁拥有语义，你也无法稳定复现时序问题。

我的判断是：

1. **props/event 是默认解**，因为它们把依赖方向写在结构里，把所有权写在接口里。
2. **bus 是最后的基础设施工具**，它应该尽量少、尽量稳定、尽量无业务语义。
3. **store 是共享状态的治理中心**，但它只有在“模块边界 + 变更协议”同时成立时才会降低复杂度，否则只是把耦合集中。
4. **通信设计的目标不是“少写代码”，而是“减少不可推断性”**。可推断性才是大型项目最稀缺的资产。

当你用这套边界感去审视通信选择，你会发现很多争论会自然消失：不是“用不用 bus”，而是“这条依赖关系是否应该存在”；不是“要不要上 Vuex”，而是“这个状态是否真的是领域共享状态”。这才是 Vue 2 在大型工程里还能稳的根本原因。
